### 文档目录分层

- 别名: doc 分类, 文档落位
- 类型: 文档组织风格
- 示例: `doc/README.md`、`doc/1-架构/1-总架构.md`、`doc/1-架构/2-目录树.md`、`doc/1-架构/3-模块职责.md`、`doc/1-架构/4-主要业务链路.md`、`doc/2-需求/README.md`、`doc/3-实施/README.md`、`doc/4-bugs/README.md`、`doc/5-tests/README.md`、`doc/6-审查/README.md`、`doc/7-验收/README.md`
- 说明: 根 `doc/README.md` 负责总导航；`doc/1-架构/` 固定使用 `1-4` 四个稳定主入口承接长期架构专题，其他专题使用 `附录-<中文主题>.md`，并且在目录规则更新后也仍按正式长期目录处理，不降级成历史目录；需求主文档和配套 SVG 放入 `doc/2-需求/`；实施目录导航默认先区分“阶段导航主入口”“按需求归类的实施总览”“按阶段归类的实施文档”；Bug 记录放入 `doc/4-bugs/`，测试验证资产放入 `doc/5-tests/`，审查结论与目录归位说明放入 `doc/6-审查/`，验收标准与最终验收放入 `doc/7-验收/`。当用户对目录职责产生疑问时，默认以最新 `AGENTS.md`、`artifact-storage-rules` 和当前审查结论文档复核；若复核结果只是“正式落点有效，导航需再收口”，则新增审查归档并更新入口，不重复迁移正式文档。
- 来源: `AGENTS.md`; `doc/README.md`; `doc/1-架构/README.md`; `doc/1-架构/1-总架构.md`; `doc/1-架构/2-目录树.md`; `doc/1-架构/3-模块职责.md`; `doc/1-架构/4-主要业务链路.md`; `doc/2-需求/README.md`; `doc/3-实施/README.md`; `doc/7-验收/README.md`; `doc/4-bugs/README.md`; `doc/5-tests/README.md`; `doc/6-审查/README.md`; `doc/6-审查/2026-06-28_按更新后skill复核截图文档归类与导航收口.md`; `doc/6-审查/2026-06-28_文档目录最终收口审计.md`

### 架构边界矩阵写法

- 别名: boundary matrix, 不负责事项, 专题速查
- 类型: 架构文档风格
- 示例: 在 `doc/1-架构/README.md` 先列“控制配置层三件套”“战斗载体三件套”速查表；在具体附录里再补“分层边界速查”与“本文不负责的事项”，例如把首页附录限制在首页主入口与首页预览展示包，把模块索引附录限制在继续上次/开始新游戏与解锁规则，把状态效果附录限制在 `effects/<effect_id>/` 生命周期与叠层驱散；若正文里已经有速查表，则删除相邻章节里重复的长段职责解释。
- 说明: 当多个长期附录容易互相引用并重复描述职责时，优先新增“边界矩阵 + 不负责事项”，再把正文里的重复职责段落压缩掉，而不是先新建更多平行附录。这样可以在不扩大文档数量的前提下，把高重叠专题的真相源边界钉死。
- 来源: `doc/1-架构/README.md`; `doc/1-架构/附录-首页预览展示包与首页入口规范.md`; `doc/1-架构/附录-模块索引与启动入口规范.md`; `doc/1-架构/附录-模块正文配置与模块名单规范.md`; `doc/1-架构/附录-技能配置与战斗表现规范.md`; `doc/1-架构/附录-武器投射物与命中效果规范.md`; `doc/1-架构/附录-状态效果与属性修饰规范.md`
- 适用范围: `doc/1-架构/`
- 更新时间: 2026-06-28
- 状态: 启用

### 控制配置层三件套正文去重写法

- 别名: 首页问题回首页附录, 入口问题回模块索引附录, 正文问题回模块正文附录
- 类型: 架构文档风格
- 示例: 首页附录删除 `continue_entry_policy` 和继续上次策略描述，只保留首页主入口、公共导航和首页预览展示包；模块索引附录删除 `module.yaml` 最小字段契约，改为只引用模块正文附录；模块正文附录删除 `modules_index.yaml` 字段展开，只把它保留为相邻入口层边界引用。
- 说明: 当控制配置层三件套已经有总表，但正文里仍互相解释对方的载体时，优先继续在正文层做去重，而不是再新增“补充说明”文档。默认规则是：首页问题回首页附录，入口问题回模块索引附录，正文与名单问题回模块正文附录。

### 旧文档承接映射写法

- 别名: 历史专题映射表, old-to-new doc mapping
- 类型: 文档组织风格
- 示例: 当用户基于截图或历史 dated 文档要求重新归类时，优先在正式目录入口补一张“历史文档 -> 当前正式入口”的映射表，例如在 `doc/1-架构/README.md` 增加 `2026-06-22_*` 到 `附录-*` 的承接映射，并在 `doc/2-需求/README.md`、`doc/3-实施/README.md` 增加“截图旧需求承接”“截图旧实施承接”速查，而不是只在审查文档里口头说明。
- 说明: 这种写法让正式入口本身就能回答“旧文档现在看哪一份”，详细逐项说明再保留到审查文档，减少在多份审查说明之间来回查找的成本。
- 来源: `doc/1-架构/README.md`; `doc/6-审查/2026-06-28_截图文档目录规则复核与最终归档.md`
- 适用范围: `doc/1-架构/`; `doc/README.md`; `doc/6-审查/`
- 更新时间: 2026-06-28
- 状态: 启用

### 截图逐项对照表写法

- 别名: screenshot file mapping table, 旧文件逐项归类总表
- 类型: 文档组织风格
- 示例: 当用户直接给一张目录截图，希望按职责重新归类时，优先在 `doc/6-审查/` 新增一份“截图文件 -> 当前归属目录 -> 当前正式入口 -> 处理方式 -> 说明”的逐项对照表，例如 `doc/6-审查/2026-06-28_截图文档逐项归类对照表.md`，并把它加入 `doc/README.md` 与 `doc/6-审查/README.md` 的当前优先入口，而不是把同日多份整理说明全部挂进导航。
- 说明: 这种写法适合“旧文件较多、部分保留历史来源、部分并入新正式入口”的整理场景。它比单纯写一段归类说明更直观，也能减少后续继续拿截图逐条追问时的沟通成本；导航层只要能把人直接带到这张总表即可。
- 来源: `doc/6-审查/2026-06-28_截图文档逐项归类对照表.md`; `doc/6-审查/2026-06-28_文档索引入口与截图归类导航再收口.md`; `doc/README.md`; `doc/6-审查/README.md`
- 适用范围: `doc/6-审查/`; 文档目录整理场景
- 更新时间: 2026-06-28
- 状态: 启用

### 正式入口与审查归档分层写法

- 别名: 正式正文与整理过程分层, final doc classification writeup
- 类型: 文档组织风格
- 示例: 当目录归类已经完成后，`doc/README.md` 只保留目录职责、当前正式入口和最小回查入口；“这轮怎么整理、为什么这样收口、哪些说明应只保留在归档”单独落到 `doc/6-审查/2026-06-28_文档目录职责归类与正式入口最终归档.md`、`doc/6-审查/2026-06-28_文档索引入口与截图归类导航再收口.md`、`doc/6-审查/2026-06-28_按更新后skill复核截图文档归类与导航收口.md`、`doc/6-审查/2026-06-29_doc总导航再次压回目录职责与最小回查入口.md` 这类审查文档；`doc/6-审查/README.md` 也只保留少量当前优先入口。
- 说明: 这种写法可以避免正式入口继续堆积历史整理过程，也避免把审查说明重新混入需求、实施或架构正文。默认边界是：正式入口承接业务正文，审查目录承接整理过程与规则复核，若本轮只是按新规则复核并发现正式落点已正确，则只新增审查结论并更新导航，不重新制造平行业务文档。
- 来源: `doc/README.md`; `doc/6-审查/2026-06-28_文档目录职责归类与正式入口最终归档.md`; `doc/6-审查/2026-06-28_文档索引入口与截图归类导航再收口.md`; `doc/6-审查/2026-06-28_按更新后skill复核截图文档归类与导航收口.md`; `doc/6-审查/README.md`
- 适用范围: `doc/`; `doc/6-审查/`; 文档目录整理场景

- 状态: 启用

### 目录首页标题统一收紧写法

- 别名: 当前状态改目录现状, 首页标题去阶段性残留
- 类型: 文档组织风格
- 示例: 在 `doc/README.md` 的目录结构表中，把 `当前状态` 收紧为 `目录现状`；在 `doc/4-bugs/README.md`、`doc/5-tests/README.md`、`doc/7-验收/README.md`、`game/README.md`、`tools/windows/README.md` 中，把 `## 当前状态` 统一改成 `## 目录现状`。
