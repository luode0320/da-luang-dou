# 地图分层与区域机制规范

## 目标

本附录用于把阶段 3 中已经提出、但还没有在长期架构层写硬的“地图分层、区域定义、碰撞边界、刷怪区域、事件区域、机关与天气机制、地图预览和引用关系”统一成长期契约。

它重点回答以下问题：

- `maps/<map_id>/` 目录长期应该如何分层
- `config.yaml`、`tiles/`、`props/`、`zones/` 和 `overrides/` 各自负责什么
- 地图层、区域层、碰撞层、刷怪区和事件区最少应该有哪些字段
- 模块地图、关卡配置、怪物刷怪表和事件机制如何共用同一套地图 ID 与区域 ID 口径
- 后台与 Godot 运行时最少要校验哪些地图结构问题

## 一、分层总原则

### 1. 地图不是单张背景图，而是可分层的可玩场景单元

- 地图长期必须至少区分底图、装饰、交互、碰撞和区域语义。
- 地图正文负责“空间结构和可玩边界”。
- 关卡正文只负责“在这张地图上发生什么关卡逻辑”，不复制地图空间结构。

### 2. 地图目录只承载地图本体，不替代关卡、怪物和事件正文

- 地图目录负责：地图基础信息、素材分层、区域、碰撞、刷怪区、事件区和机制入口。
- 关卡目录负责：关卡时长、节点推进、波次和奖励。
- 怪物目录负责：怪物实体与技能表现。
- 事件或规则本体若复杂，应由规则包或事件配置承接，地图只保留入口引用。

### 3. 地图引用必须基于稳定 ID，而不是临时路径或场景名

- 关卡引用地图时只引用 `map_id`。
- 刷怪、事件、关卡或调试入口引用区域时只引用稳定 `zone_id`、`spawn_zone_id`、`event_zone_id`。
- 不允许在关卡或刷怪表里直接写散乱 Godot 节点路径替代长期配置 ID。

## 二、目录与载体

长期建议至少采用以下结构：

```text
game/data/modules/<module_id>/maps/<map_id>/
├── config.yaml
├── assets/
├── tiles/
├── props/
├── zones/
└── overrides/
```

说明：

- `config.yaml` 负责地图主配置。
- `assets/` 负责地图总素材，如缩略图、背景图、预览图。
- `tiles/` 负责瓦片、地表、地形与底图资源。
- `props/` 负责场景装饰、障碍、道具摆件和交互道具。
- `zones/` 负责可移动区、禁行区、碰撞区、刷怪区、事件区和出生点定义。
- `overrides/` 负责模块级局部覆盖，不应成为复制整张地图的兜底目录。

## 三、地图主配置最小契约

### 3.1 `config.yaml` 最小字段

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `id` | 地图稳定 ID |
| `module_id` | 所属模块 |
| `name_zh` | 中文显示名 |
| `theme` | 地图主题 |
| `background_asset_id` | 背景素材 |
| `tile_pack_id` | 瓦片资源包 |
| `prop_pack_id` | 场景道具包 |
| `zone_pack_id` | 区域定义包 |
| `spawn_pack_id` | 刷怪区域定义包 |
| `event_pack_id` | 事件区域定义包 |
| `weather_pack_id` | 天气配置 |
| `lighting_pack_id` | 光照配置 |
| `preview_asset_id` | 地图缩略图或预览图 |

### 3.2 主配置字段约束

- `id`、`module_id`、`theme`、`tile_pack_id`、`zone_pack_id` 必填。
- `background_asset_id` 可以为空，但不能以“缺背景就把地图降级成空白”替代真实校验。
- `spawn_pack_id` 与 `event_pack_id` 至少允许为空表，但若为空表则要显式可识别。
- `preview_asset_id` 建议必填，用于后台和游戏侧预览。

## 四、地图层最小分层

长期建议至少区分以下层：

| 层级 | 职责 | 典型内容 |
| --- | --- | --- |
| `base` | 底图与地表 | 地砖、地板、地形底色 |
| `decor` | 装饰层 | 树木、石头、场景摆件 |
| `interactive` | 交互层 | 机关、可触发物、障碍物 |
| `collision` | 碰撞层 | 不可穿越边界、遮挡物 |
| `zone` | 区域语义层 | 出生点、刷怪区、事件区、可移动区 |

统一约束：

- 地图至少应有 `base`、`collision`、`zone` 三层。
- `decor` 缺失可以接受，但不能因此把 `interactive` 或 `collision` 混回底图。
- `zone` 层必须可被发布配置和运行时共同引用。

## 五、区域定义最小契约

### 5.1 `zones/` 建议内容

建议至少支持以下区域类型：

- `spawn_point`
- `walkable_zone`
- `blocked_zone`
- `spawn_zone`
- `event_zone`
- `boss_zone`

### 5.2 单个 zone 最小字段

建议每个区域至少包含：

| 字段 | 作用 |
| --- | --- |
| `zone_id` | 区域稳定 ID |
| `zone_type` | 区域类型 |
| `shape` | 几何形状，如矩形、多边形、点 |
| `coordinates` | 几何坐标 |
| `tags` | 标签 |
| `enabled` | 是否启用 |
| `notes` | 备注或用途说明 |

### 5.3 区域约束

- `zone_id` 在同一地图内不得重复。
- `spawn_zone` 必须能被刷怪表引用。
- `event_zone` 必须能被事件机制或关卡触发引用。
- `walkable_zone` 与 `blocked_zone` 的组合应能解释玩家可移动边界。

## 六、碰撞与可移动边界规则

### 6.1 碰撞层职责

- 负责定义不可穿越区域。
- 负责定义障碍、墙体、封闭边界和限制移动的地形。
- 不负责刷怪和事件逻辑。

### 6.2 可移动边界约束

- 玩家出生点必须位于合法可移动区域内。
- 刷怪区可以与可移动区部分重叠，但不能完全落在不可达封闭区里，除非玩法明确需要。
- 若 Boss 区域有阶段切换或封门逻辑，也应先以区域和碰撞语义表达，再由运行时行为承接。

## 七、刷怪区与事件区规则

### 7.1 刷怪区

- 刷怪区是地图语义，不是怪物波次正文。
- 刷怪表通过 `spawn_zone_id` 引用地图刷怪区。
- 一个关卡可以引用多个刷怪区，但引用口径必须统一。

### 7.2 事件区

- 事件区负责表达机关、天气触发、剧情触发、区域惩罚或区域限制的空间入口。
- 事件本体参数应由规则包或事件配置承接；地图只提供区域锚点和事件引用入口。

### 7.3 Boss 区域

- Boss 区域建议显式作为区域类型或标签保留。
- Boss 关卡引用 Boss 区域时，仍然通过稳定区域 ID，而不是直接写 Godot 场景节点名。

## 八、地图机制与预览规则

### 8.1 地图机制

长期建议地图层至少允许表达以下机制入口：

- 机关
- 天气
- 光照
- 地形限制
- 区域事件

统一约束：

- 机制入口可以放在地图主配置中，但复杂机制参数不应全部堆在地图主配置里。
- 地图主配置更适合引用机制包或机制 ID。

### 8.2 地图预览

- 后台和游戏侧应至少能读取 `preview_asset_id` 作为缩略图。
- 若存在多层预览，可进一步区分“后台编辑缩略图”和“游戏侧展示缩略图”，但不另造一套地图主 ID。

## 九、与关卡、怪物和事件的引用关系

统一建议按以下关系引用：

1. 关卡通过 `map_id` 引用地图
2. 波次通过 `spawn_zone_id` 引用地图刷怪区
3. 事件规则通过 `event_zone_id` 引用地图事件区
4. Boss 关通过 `boss_zone` 或 `zone_id` 标签引用地图 Boss 区域

不允许：

- 关卡直接写地图内部散乱路径替代 `map_id`
- 刷怪表直接写临时坐标字符串代替长期区域 ID
- 事件表直接把区域信息复制进多份关卡配置

## 十、发布前最小校验集合

长期建议发布前至少校验以下内容：

| 检查项 | 最小要求 | 阻断条件 |
| --- | --- | --- |
| 地图主配置完整性 | `id`、主题、瓦片包、区域包可解析 | 关键字段缺失 |
| 地图层完整性 | 至少存在底图、碰撞、区域语义 | 地图退化为单张背景图 |
| 出生点合法性 | 出生点位于可移动区域 | 出生点落在不可达区 |
| 刷怪区引用合法性 | 刷怪区可被波次引用 | 刷怪区不存在 |
| 事件区引用合法性 | 事件区可被事件或关卡引用 | 事件区不存在 |
| 关卡地图引用 | 关卡中的 `map_id` 可定位 | 地图引用断链 |
| 区域 ID 唯一性 | 区域 ID 无冲突 | 同图重复 ID |

## 十一、运行时最小输出字段

地图加载、关卡进入和区域触发时，运行时最少应输出：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 当前模块 |
| `map_id` | 当前地图 |
| `theme` | 当前主题 |
| `spawn_zone_id` | 当前刷怪区 |
| `event_zone_id` | 当前事件区 |
| `preview_asset_id` | 当前预览图 |
| `collision_state` | 当前碰撞加载结果 |
| `zone_pack_id` | 当前区域定义包 |

## 十三、结论

地图配置长期最容易失控的，不是“素材有没有导进去”，而是下面三层边界有没有先写硬：

- 地图本体分层负责什么
- 区域与碰撞语义负责什么
- 关卡、刷怪和事件如何通过稳定 ID 引用地图

只要这三层先写硬，后续 Godot 地图构建、后台地图配置、关卡引用和发布校验就能围绕同一套长期地图契约推进。
