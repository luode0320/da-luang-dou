# 发布契约层阶段0蓝图再次压回最小闭环与依赖关系

## 背景

在发布契约层连续收口后，`附录-阶段0配置底座落地蓝图.md` 仍保留了一段与 `附录-发布包最小字段契约.md` 高度重合的正文：

- 直接重复发布目录树
- 直接重复逐步读取顺序
- 在最小闭环和最小蓝图中继续点名 `manifest.yaml`、`compatibility.yaml` 与三份索引

这会让阶段 0 蓝图重新承担字段层真相源，而不是停留在“阶段 0 为什么必须这样闭环、闭环后给后续阶段提供什么依赖”的长期蓝图职责。

## 本轮目标

- 把阶段 0 蓝图继续压回“最小闭环 + 最小蓝图 + 后续依赖关系”。
- 把具体目录树、文件名、字段清单和精确读取顺序继续转交最小字段契约。

## 本轮实际调整

### 最小闭环表达收紧

- 把“发布包至少具备 `manifest.yaml`、`compatibility.yaml` 和三份最小索引”收紧为“发布版本入口、兼容契约入口和最小索引入口集合”。
- 把后台发布层从“生成 `manifest`、`compatibility`、索引和标准化正文目录”收紧为“生成发布版本入口、兼容契约入口、索引入口和标准化正文目录”。
- 把游戏读取器最小蓝图中的显式文件读取步骤改成入口类型表达。

### 发布包最小实现闭环收紧

- 删除蓝图文档里与最小字段契约重复的发布目录树。
- 删除蓝图文档里与最小字段契约重复的精确读取顺序。
- 改写为六条抽象闭环要求：
  - 版本化发布目录
  - 发布版本入口
  - 兼容契约入口
  - 索引入口集合
  - 正文配置目录
  - 统一读取到合并缓存追踪的顺序

## 边界结论表

| 文档 | 本轮收紧后负责 | 不再负责 |
| --- | --- | --- |
| `附录-阶段0配置底座落地蓝图.md` | 阶段0长期定位、最小闭环、后台/游戏最小蓝图、后续阶段依赖关系 | 具体目录树、字段清单、精确读取顺序 |
| `附录-发布包最小字段契约.md` | 发布版本入口、兼容契约入口、索引入口、正文目录的具体结构与读取顺序 | 阶段0为什么优先、对后续阶段的依赖输出 |

## 结论

这轮之后，阶段 0 蓝图再次回到“长期蓝图”定位：它说明阶段 0 最少必须打通什么、为什么这是后续阶段共同底座，但不再平行复制字段契约文档里的具体文件结构和读取步骤。
