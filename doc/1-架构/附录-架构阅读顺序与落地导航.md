# 架构阅读顺序与落地导航

## 目标

本附录不新增业务规则，只负责回答两个问题：

1. 当前项目的长期架构文档，后续应该按什么顺序阅读。
2. 当开始落地阶段 0、阶段 1-7 或具体模块时，优先以哪些文档为准。

这样可以避免 `1-总架构.md`、`项目设计.md`、阶段文档和各附录同时承担“导览说明”职责。

统一约束：

- 本附录是“先读什么、后读什么、开始实现前该先看哪组文档”的唯一顺序入口。
- `1-总架构.md` 只负责入口分工与分层摘要，不再并行承担阅读顺序说明。
- `doc/1-架构/README.md` 只负责目录导航与专题分组，不再并行承担阅读顺序说明。

## 适用范围

- 新成员或后续 AI 继续接手本项目时的预热入口
- 开始写 Go 后端、Vue 后台或 Godot 读取器前的架构预读
- 需要判断“这个问题该先看哪份文档”时的导航入口

## 一、先读哪几层

推荐固定按以下四层阅读：

### 第 1 层：项目总览层

先读：

- [项目设计](../../项目设计.md)

这一层只回答：

- 项目是什么
- 当前做到什么阶段
- 产品与玩法的顶层边界是什么

### 第 2 层：架构总入口层

再读：

- [1-总架构](1-总架构.md)
- [2-目录树](2-目录树.md)
- [3-模块职责](3-模块职责.md)
- [4-主要业务链路](4-主要业务链路.md)

这一层只回答：

- 长期目录和模块怎么分层
- 主链路有哪些
- 专题附录应该从哪里进入

### 第 3 层：阶段 0 配置底座层

如果开始落地实现，优先读：

- [附录-阶段0配置底座落地蓝图](附录-阶段0配置底座落地蓝图.md)
- [附录-发布包最小字段契约](附录-发布包最小字段契约.md)
- [附录-后台发布器与游戏读取器目标分层](附录-后台发布器与游戏读取器目标分层.md)
- [附录-游戏侧配置读取与合并规范](附录-游戏侧配置读取与合并规范.md)

这一层只回答：

- 阶段 0 最小闭环是什么
- 后台发布器和游戏读取器各自最少要实现什么
- 发布包和运行时读取的最小契约是什么

### 第 4 层：专题附录层

最后按具体问题进入相应专题：

- 首页与入口问题：先看首页附录、模块索引附录、模块正文附录
- 战斗载体问题：先看技能附录、武器投射物附录、状态效果附录
- 发布链路问题：先看发布契约层六件套
- 存档恢复与验证入口问题：先看存档生命周期附录、调试入口附录和 `9-业务链路`

## 三、哪些文档不该再承担导航职责

- `项目设计.md` 不再负责展开“先读哪份附录”的阅读顺序。
- `1-总架构.md` 不再重复平铺“阶段 0 最小闭环怎么落地”的细节。

## 四、结论

- 后续预热项目时，默认先看这份导航，再按层进入。
- 这份文档的价值不是新增规则，而是减少入口重复解释。
- 如果后续附录继续增加或主链路重排，优先更新本文的阅读顺序，而不是同时改动多个总入口去重复写导览说明。
