## 项目目标与范围

- 项目目标：打造一款面向 Steam/PC 的 2D 自动战斗闯关游戏，关卡推进参考《植物大战僵尸》的模块化章节结构，战斗节奏参考《土豆兄弟》的自动输出与持续清怪体验。
- 第一阶段目标：先形成“继续上次 / 开始新游戏 -> 进入模块 -> 连续闯关 -> 掉落模块内临时道具 -> 保存进度与长期资产”的最小可迭代 Demo。
- 当前不覆盖范围：复杂图鉴、成就系统、复杂商店、多模式入口和大规模运行时热更新，不作为第一阶段闭环前提。

## 技术栈与运行形态

- 主游戏：Godot 4，工程目录为 `game/`。
- 后台系统：Go 后端 + Vue 前端，目录为 `admin/backend/` 与 `admin/frontend/`。
- 配置真源：后台发布后的 YAML 配置包，游戏侧只读 `game/data/published/` 下的发布结果。
- Windows 分发：后续打包、安装器和发布辅助脚本统一归 `tools/windows/`。
- 素材工作流：正式 2D 素材先通过 `2d-asset-design` / `imagegen` 形成设计稿，再进入 Godot 资产接入流程。

## 核心架构方向

### 1. 配置驱动优先

- 长期扩展内容优先由配置驱动，不把模块、关卡、角色、怪物、地图、技能、道具和掉落规则写死在业务代码里。
- 具体配置层拆分、目录边界和字段专题，统一转交 `doc/1-架构/`。

### 2. 三态单向流转

- 项目长期坚持“后台编辑态 -> 发布契约态 -> 游戏运行态”的单向流转。
- 三态边界、发布契约和读取细节，统一转交架构主入口与发布专题附录。

### 3. 阶段 0 先行

- 阶段 0 是后续首页、模块、地图、角色、怪物、关卡和发布验证的共同地基。
- 具体实现蓝图与长期技术约束，统一以 `doc/1-架构/附录-阶段0配置底座落地蓝图.md` 为准。

### 4. 文档职责分层

- `项目设计.md` 负责项目总览。
- `doc/1-架构/` 负责长期架构规则与专题约束。
- `doc/3-实施/` 负责实施步骤、验证点、风险与阻断。
- `doc/5-tests/`、`doc/7-验收/` 分别负责测试证据和最终放行结论。
