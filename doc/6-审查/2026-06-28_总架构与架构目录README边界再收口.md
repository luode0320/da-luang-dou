# 总架构与架构目录README边界再收口

## 背景

- 在完成目录树、模块职责和主链路层去重后，架构目录仍存在一层总入口重叠：
  - `1-总架构.md` 在做架构分层摘要的同时，也保留了一部分目录导航语义。
  - `doc/1-架构/README.md` 在做目录导航的同时，也容易回吸“总入口摘要”。
- 如果不继续收紧，后续仍会出现“总架构页”和“目录首页”都在解释同一套阅读入口的问题。

## 本轮收口目标

- 让 `1-总架构.md` 只承担“架构域总入口”职责。
- 让 `doc/1-架构/README.md` 只承担“架构目录首页”职责。
- 把阅读顺序细节继续交给 `附录-架构阅读顺序与落地导航.md`。

## 本轮调整结果

### `1-总架构.md`

- 保留：
  - 文档说明
  - 本文负责什么 / 本文不负责什么
  - 导航分工
  - 长期架构分层摘要
  - 架构主入口矩阵
  - 与相邻文档的关系
  - 待同步项
- 删除或压缩：
  - 与目录首页重复的“目录导航式入口推荐”
  - 与 README 重复的“高重叠专题入口速查”
  - 与导览附录重复的“阅读顺序说明”

### `doc/1-架构/README.md`

- 新增：
  - 文档说明
  - 本页负责什么 / 本页不负责什么
- 保留：
  - 先看哪一组
  - 稳定主入口
  - 业务链路入口
  - 阶段 0 与导览入口
  - 高重叠专题速查
  - 其他专题入口
  - 历史专题承接映射
  - 落点规则
- 明确它不再承担：
  - 项目总览
  - 长期架构分层摘要
  - 待同步项

## 当前分工结论

| 文档 | 现在主要回答的问题 | 不再回答的问题 |
| --- | --- | --- |
| `1-总架构.md` | 长期架构层与项目总览层、实施层如何分工；架构域主入口矩阵是什么；还有哪些待同步项 | 架构目录里有哪些分组入口、旧文档映射表、专题速查表 |
| `doc/1-架构/README.md` | 架构目录应该从哪里开始看；专题文档分别在哪一组；旧文档现在应该看哪一份 | 长期架构层本身的分层摘要、待同步项和顶层架构边界总结 |

## 后续维护规则

- 若变化属于“架构层级关系、主入口矩阵、待同步项”，优先更新 `1-总架构.md`。
- 若变化属于“目录导航、专题分组、历史专题承接映射、入口速查”，优先更新 `doc/1-架构/README.md`。
- 若变化只是“阅读顺序细节”，优先更新 `附录-架构阅读顺序与落地导航.md`，不要同时改总架构页和目录首页。

## 结论

- 本轮继续把总入口层拆成“架构域总入口”和“架构目录首页”两层。
- 收口后：
  - `1-总架构.md` 更接近“分层摘要 + 主入口矩阵 + 待同步项”
  - `doc/1-架构/README.md` 更接近“目录导航 + 专题速查 + 历史映射”
- 后续若这两份又开始互相吸收内容，默认按“层级问题回总架构，目录问题回 README，阅读顺序问题回导览附录”的规则继续收紧。
