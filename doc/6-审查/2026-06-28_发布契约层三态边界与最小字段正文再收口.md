# 发布契约层三态边界与最小字段正文再收口

## 背景

- 发布契约层六件套最近几轮已经持续去重，其中：
  - `附录-发布包最小字段契约.md` 已收紧为字段与读取入口真相源
  - `附录-版本矩阵与兼容判定规范.md` 已收紧为版本对象与判定口径真相源
- 继续复核后，`附录-发布包契约与三态边界.md` 里仍保留一段较细的“建议目标结构”，与最小字段契约的目录结构正文形成残留重叠。

## 本轮收口目标

- 让“三态边界”继续只讲层、职责与单向流转。
- 让“最小字段契约”继续只讲版本目录、正式文件、索引和最小读取顺序。
- 避免三态文档再承担接近字段级目录结构说明书的职责。

## 本轮实际调整

### `附录-发布包契约与三态边界.md`

- 删除较细的“建议目标结构”树形展开。
- 改为只保留结构层最小语义：
  - 运行时唯一正式输入根目录长期收口到 `game/data/published/`
  - 每个发布版本都必须是独立、只读、可追踪的版本化目录
  - 版本目录内部至少要有“版本入口 + 兼容入口 + 索引入口 + 正文配置入口”四类载体
- 明确把这些载体的具体目录、文件名和最小字段继续转交 `附录-发布包最小字段契约.md`。

## 本轮边界结论

| 文档 | 本轮收紧后的唯一正文职责 | 不再展开 |
| --- | --- | --- |
| `附录-发布包契约与三态边界.md` | 三态职责边界、长期唯一输入根、单向流转、结构层最小语义 | 具体目录树、文件名清单、字段表 |
| `附录-发布包最小字段契约.md` | 发布版本目录、正式文件、索引、最小字段与最小读取顺序 | 三态语义、版本判定口径 |

## 结论

- 这轮收口后，`三态边界` 与 `最小字段契约` 的正文分工更硬：
  - 三态文档只讲“层与流转”
  - 字段文档只讲“结构、字段、读取入口”
- 后续若继续压发布契约层重叠点，优先再检查 `附录-发布包契约与三态边界.md` 与 `5-业务链路-后台配置发布与游戏读取.md` 之间，是否还残留对相同主流程的平铺解释。
