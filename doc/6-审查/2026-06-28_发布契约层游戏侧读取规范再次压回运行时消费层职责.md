# 发布契约层游戏侧读取规范再次压回运行时消费层职责

## 背景

`附录-游戏侧配置读取与合并规范.md` 仍保留旧版规范写法，主要问题不是字段重复很多，而是正文边界没有被正式钉死：

- 它没有明确声明自己与“最小字段契约”“目标分层”的分工。
- 末尾总链路表达仍容易让它看起来像在承接入口读取顺序。
- 错误处理、降级和调试输出也没有和相邻发布专题区分清楚。

## 本轮目标

- 把游戏侧读取规范压回“运行时消费层职责”。
- 让入口、顺序、字段、组件分层继续回到相邻专题承接。

## 本轮实际调整

### 边界矩阵补齐

- 新增“分层边界速查”。
- 明确本文只负责：
  - 运行时消费职责
  - 合并规则
  - 默认值策略
  - 缓存策略
  - 热重载策略
  - 错误与运行时追踪

### 正文表达收紧

- 把“读取时机”收紧为“消费已定位好的正式配置结果的时机”。
- 把结论链路从“读取发布包 -> 解析索引 ...”收紧为“消费正式配置入口 -> 定位正文 ...”。
- 在错误处理里补充“降级只发生在运行时允许降级的字段或资源上”，并把通过 / 警告 / 阻断继续转交版本矩阵与发布验证专题。

### 文档边界补齐

- 新增“与现有文档的关系”。
- 新增“本文不负责的事项”。
- 明确：
  - 字段清单与精确读取顺序回到 `附录-发布包最小字段契约.md`
  - 组件层级与命名回到 `附录-后台发布器与游戏读取器目标分层.md`
  - 兼容判定与放行结论回到相邻发布专题

## 边界结论表

| 文档 | 本轮收紧后负责 | 不再负责 |
| --- | --- | --- |
| `附录-游戏侧配置读取与合并规范.md` | 游戏运行时消费正式发布结果后的合并、默认值、缓存、热重载、错误输出和追踪 | 精确读取顺序、字段清单、组件分层、放行结论 |
| `附录-发布包最小字段契约.md` | 发布版本入口、兼容契约入口、索引入口、正文目录和精确读取顺序 | 运行时缓存、默认值、热重载与错误追踪策略 |
| `附录-后台发布器与游戏读取器目标分层.md` | 契约读取层、合并层、缓存层、追踪层的实现层级与组件职责 | 运行时字段兜底、热重载边界和默认值策略细则 |

## 结论

这轮之后，游戏侧读取规范重新回到运行时消费层定位：它回答“拿到正式发布结果后，运行时怎么安全地合并、缓存、追踪和降级”，不再作为字段入口或代码层级的平行真相源。
