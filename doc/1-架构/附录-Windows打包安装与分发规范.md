# Windows打包安装与分发规范

## 目标

本附录用于把当前分散在阶段 7、`tools/windows/README.md` 和项目总览里的 Windows 发布要求，收口成一份长期有效的架构规范。

它重点回答以下问题：

- Godot Windows 导出产物与配置发布包如何绑定
- 哪些内容属于可执行程序，哪些内容属于配置发布结果
- 安装器、快捷方式、版本号、渠道和发布说明应该如何分层
- `tools/windows/` 长期应该承载什么脚本与辅助能力
- Windows 本地验证、安装验证、升级覆盖、回滚和卸载应满足哪些最小边界

## 一、分层总原则

### 1. 可执行程序与配置发布包分层

- Windows 构建产物负责承载 Godot 运行时与启动入口。
- 配置发布包负责承载 `manifest.yaml`、`compatibility.yaml`、索引文件和运行时配置正文。
- 不允许把后台草稿、编辑态页面结构或未发布配置直接打进正式 Windows 安装包。
- 正式安装包启动时，游戏长期只读取版本化发布包，不直接读取后台草稿。

### 2. 打包属于发布支持层，不属于游戏主逻辑层

- 打包、安装器、快捷方式、渠道参数和发布说明归 `tools/windows/` 长期维护。
- Godot 场景、战斗逻辑、角色逻辑和怪物逻辑不放进 `tools/windows/`。
- 后台发布器负责生成可运行的发布包，Windows 打包流程负责把“可执行程序 + 指定发布包”组合成交付物。

### 3. Windows 分发必须可追踪版本组合

- 任何一个 Windows 交付物，都必须能追踪：
  - `game_version`
  - `release_version`
  - `config_version`
  - `build_channel`
- 不允许只有 exe 版本号，没有配置版本或发布版本对应关系。

## 二、交付物分层

### 2.1 最小交付集合

一个可交付的 Windows 版本，最少应包含以下对象：

1. Windows 可执行程序
2. 对应的发布包目录或打包后等价只读配置载体
3. 安装器或标准安装包
4. 快捷方式入口
5. 版本与渠道元信息
6. 发布说明

### 2.2 交付对象职责

| 对象 | 职责 | 不负责 |
| --- | --- | --- |
| `game.exe` 或等价主程序 | 启动 Godot 运行时，读取指定发布包 | 存放后台编辑态数据 |
| 发布包 | 提供版本化只读配置输入 | 存放运行时日志和玩家存档 |
| 安装器 | 负责安装路径、快捷方式、卸载注册和可选升级覆盖 | 负责配置合法性校验 |
| 快捷方式 | 提供标准启动入口 | 绕过安装目录直接指向临时文件 |
| 发布说明 | 说明当前版本、渠道、变更摘要和限制 | 替代结构化版本元数据 |

## 三、推荐目录与产物关系

### 3.1 仓库内职责边界

```text
game/                       # Godot 工程与导出配置源
game/data/published/        # 后台发布后的版本化配置包
tools/windows/              # Windows 打包、安装、分发辅助脚本
admin/releases/             # 发布记录、回滚记录、审计记录
```

### 3.2 推荐打包输出关系

```text
Windows交付物/
├── game.exe
├── data/
│   └── published/
│       └── <release_version>/
├── version/
│   ├── build-info.yaml
│   └── release-notes.md
└── shortcuts/
    └── 游戏快捷方式
```

说明：

- `data/published/<release_version>/` 表示当前交付物绑定的正式发布包。
- `build-info.yaml` 用于记录 exe 构建版本、配置版本、渠道和构建时间。
- 快捷方式只应指向当前安装目录内的正式启动入口。

## 四、版本与渠道规范

### 4.1 最小版本元信息

Windows 交付物至少应能追踪以下字段：

| 字段 | 作用 |
| --- | --- |
| `game_version` | 对应 Godot 构建版本 |
| `release_version` | 对应配置发布包版本 |
| `config_version` | 对应配置版本 |
| `build_channel` | 开发、测试、正式等渠道 |
| `build_time` | 构建时间 |
| `release_notes_ref` | 发布说明引用 |

### 4.2 渠道边界

- `dev`：开发渠道，可包含调试参数和额外日志，不作为正式玩家分发版本。
- `test`：测试渠道，用于测试环境安装、回归和放行前验证。
- `prod`：正式渠道，用于真实交付或外部演示。

统一约束：

- 不同渠道不能共用含义不清的版本号。
- 安装器、发布说明和运行时日志必须能看出当前渠道。
- 开发或测试渠道不得伪装成正式渠道安装包。

## 五、安装器与快捷方式规范

### 5.1 安装器最小职责

安装器长期至少应负责：

1. 选择安装路径
2. 写入主程序
3. 写入绑定发布包
4. 创建桌面或开始菜单快捷方式
5. 提供卸载入口

### 5.2 快捷方式规则

- 快捷方式应指向正式安装目录内的启动程序。
- 快捷方式启动后，首页应能读取当前绑定的 `release_version`。
- 不允许快捷方式仍指向临时构建目录、导出缓存目录或开发机本地路径。

## 六、升级、覆盖、回滚与卸载边界

### 6.1 升级覆盖

升级覆盖默认至少满足：

1. 可识别当前已安装版本
2. 可覆盖主程序与发布包
3. 不误删玩家存档和长期资产
4. 覆盖后仍能读取新版本配置并启动到首页

### 6.2 回滚边界

- 回滚不是单纯替换 exe；应同时考虑回滚到对应的 `release_version`。
- 回滚后应按 [附录-发布验证与放行闭环规范](附录-发布验证与放行闭环规范.md) 至少完成一次冒烟启动。
- 若目标版本与当前存档不兼容，必须显式给出阻断或降级提示，不得静默损坏存档。

### 6.3 卸载边界

- 卸载默认移除主程序、安装器注册和快捷方式。
- 是否保留玩家存档、日志和长期资产，应在安装器策略中明确。
- 若默认保留存档，必须在卸载说明中写清。

## 七、`tools/windows/` 长期职责

### 7.1 应承载的脚本类型

`tools/windows/` 后续应优先沉淀以下长期脚本：

- Godot Windows 导出辅助脚本
- 安装器构建脚本
- 渠道参数注入脚本
- 构建元信息生成脚本
- 本地安装验证脚本

### 7.2 不应承载的内容

- 游戏业务逻辑
- 后台配置编辑逻辑
- 临时人工操作说明代替自动化脚本

### 7.3 与其他目录的关系

- `tools/windows/` 读取 `game/` 导出产物，但不定义游戏业务。
- `tools/windows/` 使用 `game/data/published/` 的正式发布包，但不生成后台草稿。
- `tools/windows/` 可引用 `admin/releases/` 的发布记录与回滚记录，用于构建说明和版本追踪。

## 八、最小验证路径

### 8.1 本地构建验证

最小路径：

1. 导出 Windows 可执行程序
2. 绑定指定 `release_version`
3. 写入构建元信息
4. 本地启动到首页
5. 在首页或日志中确认 `game_version` 与 `release_version`

### 8.2 安装验证

最小路径：

1. 生成安装器
2. 执行安装
3. 创建快捷方式
4. 通过快捷方式启动到首页
5. 确认首页可读取当前发布配置

### 8.3 升级覆盖验证

最小路径：

1. 安装旧版本
2. 使用新安装包执行覆盖
3. 启动后确认版本切换成功
4. 校验存档与长期资产未被误删

### 8.4 回滚验证

最小路径：

1. 切换到目标发布版本
2. 安装或替换到目标交付物
3. 启动并完成一次冒烟链路
4. 记录回滚验证结果

## 十、结论

Windows 打包、安装与分发长期不是“最后导出一个 exe”这么简单，而是一条稳定的发布支持链路：

- 可执行程序负责运行
- 发布包负责内容
- 安装器负责交付
- 渠道与版本元信息负责追踪
- 最小验证路径负责放行

只有把这几层边界先写硬，后续 `tools/windows/`、阶段 7 验证和正式交付流程才能围绕同一套架构规则推进。
