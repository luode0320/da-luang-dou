# 配置目录规范 v1

## 目标

本规范用于统一项目中“配置如何组织、素材如何归档、后台如何引用、游戏如何读取”的目录标准。核心原则是：

- 能目录化的内容尽量目录化。
- 能和素材强绑定的内容必须目录化。
- 后台配置时优先通过“选择目录 + 选择主配置 + 自动带出素材引用”的方式完成绑定。
- 游戏侧只读取后台发布后的合并结果，不直接依赖后台编辑态结构。

## 目录化原则

### 必须使用子目录的内容

以下内容建议强制采用“实体目录 + 主配置 + 资源子目录”的组织方式：

- 首页展示包
- 角色
- 怪物
- BOSS
- 地图
- 关卡节点组
- 技能
- 特效 / VFX
- 音效 / SFX
- 动画帧资源
- 道具
- 掉落表
- 刷怪表
- 难度包
- 模块规则包
- 文案本地化包
- UI 组件预设

### 建议使用子目录的内容

以下内容可以目录化，便于后续扩展、复用和后台引用：

- 武器配置
- 子弹 / 投射物配置
- Buff / Debuff 配置
- 场景机关配置
- 关卡事件脚本配置
- 角色皮肤或变体配置
- 怪物变体配置
- 模块主题包

### 可以保持平铺的内容

以下内容通常适合保留为平铺 YAML，不必强制拆成目录：

- 全局常量
- 枚举表
- 通用倍率表
- 少量全局开关
- 一组非常小且稳定的系统参数

## 单实体目录约定

每个实体目录建议包含以下内容：

- `config.yaml`：实体主配置
- `assets/`：与该实体直接绑定的素材
- `overrides/`：覆盖参数或局部变体

地图类实体建议额外包含：

- `tiles/`：瓦片或地形相关资源
- `props/`：场景道具与装饰资源
- `zones/`：刷怪区、事件区、碰撞区等区域定义

技能类实体建议额外包含：

- `cast/`：施放阶段资源
- `impact/`：命中阶段资源
- `travel/`：飞行或移动阶段资源
- `cooldown/`：冷却表现资源

武器、投射物类实体的目录与字段细化，统一以 [附录-武器投射物与命中效果规范](附录-武器投射物与命中效果规范.md) 为准。
效果包、属性修饰、叠层与驱散类实体的目录与字段细化，统一以 [附录-状态效果与属性修饰规范](附录-状态效果与属性修饰规范.md) 为准。

## 后台引用方式

后台配置时建议采用以下顺序：

1. 选择模块或全局目录。
2. 选择实体类型。
3. 选择实体目录。
4. 选择主配置文件。
5. 自动带出该目录下可绑定素材。
6. 允许在后台编辑覆盖项并生成发布结果。

这样后台就不需要人工记路径，配置人员只需要理解“我在选哪个目录、哪个实体、哪个素材”。

## 适用边界

适合目录化的内容，一般满足至少一项：

- 需要绑定素材。
- 需要多个实例共存。
- 需要局部覆盖。
- 需要后续按模块复用。
- 需要后台树形选择。

不适合目录化的内容，一般满足以下特征：

- 只是全局固定值。
- 数量少且稳定。
- 不需要素材绑定。
- 不需要被后台按实例管理。

## 结论

本项目的配置策略应当是：

- 配置单元尽量目录化。
- 素材必须跟着实体目录走。
- 后台用目录选择来完成引用。
- 游戏侧只读取发布后的标准化结果。
