# 版本矩阵与兼容判定规范

## 目标

本附录用于把“版本矩阵”从阶段文档里的分散要求收口为长期架构规则，统一回答以下问题：

- 游戏版本、配置版本、模块版本、素材版本、存档版本分别由谁负责
- 后台发布前应该如何判断一组版本能否组成有效发布
- 游戏启动、进入模块和读取存档时应如何做兼容判定
- 回滚时什么情况允许、什么情况需要附带验证、什么情况必须阻断

本附录默认面向：

- `admin/backend/` 后续发布器与回滚器
- `game/data/published/` 下的发布包契约
- `game/` 后续配置读取器与存档恢复逻辑
- `doc/3-实施/` 阶段 0 与阶段 7 中的版本矩阵、兼容、回滚和测试验证工作

字段级载体细化见：

- [附录-发布记录回滚记录与存档元数据契约](附录-发布记录回滚记录与存档元数据契约.md)

## 一、版本对象与单一口径

版本矩阵不是把所有版本拼成一个字符串，而是让一组版本字段在“命名、来源、作用、阻断条件”上保持统一。

| 版本对象 | 统一口径 | 主要载体 | 负责方 | 主要用途 |
| --- | --- | --- | --- | --- |
| 游戏版本 | `game_version` | 构建元信息、发布记录、测试记录 | 游戏构建与发布侧 | 表示当前可执行程序的能力边界 |
| 配置版本 | `config_version` | 发布记录、版本矩阵 | 后台发布器 | 表示一组已发布内容的业务版本 |
| 发布包版本 | `release_version` | `manifest.yaml` | 后台发布器 | 表示发布包目录的运行时标识 |
| 配置契约结构版本 | `config_schema_version` | `manifest.yaml` | 后台发布器 | 表示发布包字段与目录结构版本 |
| 模块版本 | `module_version` | `indexes/modules.yaml`、版本矩阵 | 后台发布器、模块内容维护者 | 表示单模块配置快照版本 |
| 素材版本 | `asset_version` | `indexes/assets.yaml`、版本矩阵 | 后台发布器、素材流水线 | 表示素材资源快照版本 |
| 存档结构版本 | `save_schema_version` | `compatibility.yaml`、存档元数据 | 游戏运行层、后台发布器 | 表示存档结构与恢复规则版本 |

统一约束如下：

1. 业务口径统一使用 `config_version` 表示“当前发布的内容版本”。
2. 运行时发布包字段继续使用 `release_version`，但它必须与 `config_version` 一一对应，默认取同值。
3. `config_schema_version` 不等于 `config_version`。前者描述“包结构和字段契约”，后者描述“当前内容发布批次”。
4. `module_version`、`asset_version`、`save_schema_version` 都属于矩阵成员，不能只在某一处日志里口头记录。

## 二、版本载体与字段归属

### 2.1 发布记录

后台发布记录在版本矩阵中的职责，不是重新定义字段表，而是承担“业务汇总入口”。

长期要求是：发布记录必须能稳定挂接以下版本对象与结果状态：

- `game_version`
- `config_version`
- `release_version`
- `config_schema_version`
- `module_versions`
- `asset_versions`
- `save_schema_version`
- 发布结果状态与默认回滚目标

这里的发布记录是版本矩阵的业务汇总入口，适合后台界面、测试报告、发布审计和回滚说明使用。

发布记录的具体文件名、字段最小清单与落盘结构，统一转交 [附录-发布记录回滚记录与存档元数据契约](附录-发布记录回滚记录与存档元数据契约.md)。

### 2.2 manifest.yaml

`manifest.yaml` 在版本矩阵中的职责，不是重新列字段表，而是承担“运行时版本入口”。

长期要求是：它必须能稳定暴露以下版本判定所需的字段族：

- 发布包版本标识
- 发布包结构版本
- 目标游戏版本下界
- 与上一正式版本的衔接信息

统一约束：

1. `manifest.release_version` 必须与发布记录中的 `config_version` 一一对应。
2. `required_game_version` 表示“读取本发布包所要求的最小游戏版本”，不是发布记录主键。
3. `config_schema_version` 用于判断当前游戏读取器是否认识这份发布包结构。

`manifest.yaml` 的具体字段名、必填/推荐字段和文件内路径表达，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。

### 2.3 compatibility.yaml

`compatibility.yaml` 在版本矩阵中的职责，是承担“兼容判定入口”，而不是重新展开字段表。

长期要求是：它必须能稳定暴露以下兼容判定所需的字段族：

- 当前与最低支持的存档结构版本
- 明确禁止继续读取的历史版本范围
- 字段更名、字段废弃与实体 ID 重定向映射
- 当前版本的迁移说明与风险提示

统一约束：

1. 存档兼容判断必须以 `compatibility.yaml` 为准，不能只看发布记录描述。
2. 字段更名、实体 ID 更名、字段废弃等破坏性变更，必须在这里留下可执行映射或明确阻断。

`compatibility.yaml` 的具体字段名、必填/推荐字段与示例结构，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。

### 2.4 modules 与 assets 索引

长期建议把版本信息下沉到索引，以支持局部判断：

- `indexes/modules.yaml` 的每个模块项都应带 `module_version`
- `indexes/assets.yaml` 的每个素材项都应带 `asset_version`

这样后台、游戏和测试在定位问题时，能知道是“整个配置版本异常”，还是“某一个模块版本 / 素材版本异常”。

### 2.5 存档元数据

存档元数据在版本矩阵中的职责，是把“当前自动存档绑定的是哪组版本对象”稳定留下来。

长期要求是：模块自动存档元数据至少要能挂接以下恢复判断依据：

- `save_schema_version`
- `last_game_version`
- `last_config_version`
- `last_release_version`
- `module_id`
- `module_version`

这里不要求把所有素材版本写进存档，但至少要保留当前模块和当前发布版本的恢复依据。

存档元数据的具体字段名、推荐扩展字段与落盘结构，统一转交 [附录-发布记录回滚记录与存档元数据契约](附录-发布记录回滚记录与存档元数据契约.md)。

## 三、版本矩阵的三层判定

长期兼容判断默认分三层，不允许把所有问题都压成“能不能启动”一个判断。

### 3.1 构建兼容判定

判断对象：

- `game_version`
- `required_game_version`
- `config_schema_version`

结论目标：

- 当前游戏构建是否有能力读取当前发布包结构

### 3.2 内容兼容判定

判断对象：

- `config_version`
- `module_versions`
- `asset_versions`
- `field_aliases`
- `id_redirects`

结论目标：

- 当前发布内容是否能组成一份自洽的配置快照

### 3.3 存档兼容判定

判断对象：

- `save_schema_version`
- `min_supported_save_version`
- `current_save_version`
- `blocked_save_versions`
- 当前存档记录的 `last_config_version`、`module_version`

结论目标：

- 当前存档是否可以被本次发布安全读取、恢复和继续推进

## 四、发布前兼容判定规则

后台发布前默认至少执行以下规则：

| 判定项 | 规则 | 结果 |
| --- | --- | --- |
| 游戏版本下界 | `required_game_version` 高于目标 `game_version` 时，禁止发布 | 阻断 |
| 契约结构识别 | 当前目标读取器不支持 `config_schema_version` 时，禁止发布 | 阻断 |
| 配置版本映射 | `config_version` 与 `release_version` 不一致时，禁止发布 | 阻断 |
| 模块版本完整性 | `module_versions` 中任一模块无法在 `indexes/modules.yaml` 找到对应版本时，禁止发布 | 阻断 |
| 素材版本完整性 | `asset_versions` 中任一素材版本无法在 `indexes/assets.yaml` 对应到稳定素材 ID 时，禁止发布 | 阻断 |
| 破坏性字段变更 | 删除字段、更名字段、拆分关键结构但未提供 `field_aliases` / `removed_fields` 说明时，禁止发布 | 阻断 |
| 破坏性 ID 变更 | 删除实体 ID 或修改稳定 ID，但未提供 `id_redirects` 时，禁止发布 | 阻断 |
| 存档兼容范围 | 当前活跃存档版本落在 `blocked_save_versions`，且没有明确迁移路径时，禁止发布到稳定环境 | 阻断 |
| 仅素材替换 | 只替换素材内容但稳定 `asset_id` 与使用关系不变时，可继续发布，但必须进入后续发布验证链 | 通过并带验证 |
| 模块局部迭代 | 仅某个模块版本提升，其他模块未变时，可局部通过，但必须进入模块恢复与验证链 | 条件通过 |

## 五、运行时判定入口与阻断口径

游戏运行时长期应通过以下几类正式入口完成版本判定：

- 发布版本入口
- 兼容契约入口
- 模块与素材索引入口
- 当前存档元数据入口

这些入口分别用于回答：

- 当前游戏构建是否满足最低读取要求
- 当前发布包结构是否可识别
- 当前目标模块和依赖素材版本是否存在
- 当前存档是否仍在可恢复范围内

至于这些入口在运行时应该按什么精确顺序读取、正文何时进入合并与缓存阶段，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md) 与 [附录-游戏侧配置读取与合并规范](附录-游戏侧配置读取与合并规范.md)。

以下情况默认阻断读取：

- 当前游戏版本低于 `required_game_version`
- 当前读取器无法识别 `config_schema_version`
- 存档版本命中 `blocked_save_versions`
- 当前存档依赖的模块版本或关键实体 ID 已不存在，且无 `id_redirects`
- 当前存档依赖的关键字段已移除，且无 `field_aliases` 或迁移说明

以下情况允许继续，但应记录警告：

- 当前 `last_config_version` 早于最新 `config_version`，但兼容规则完备
- 当前模块版本提升，但模块入口和关卡节点仍可通过兼容映射恢复
- 素材版本提升，但逻辑素材 ID 未变

## 六、回滚判定规则

回滚不是简单切回旧目录，而是回到“某个旧版本组合”。

### 6.1 默认允许回滚

满足以下条件时可直接回滚：

- 目标 `config_version` / `release_version` 在发布记录中存在
- 当前 `game_version` 仍满足目标发布包的 `required_game_version`
- 目标 `compatibility.yaml` 明确支持当前活跃存档版本
- 当前存档涉及的模块版本与实体 ID 在目标发布包中仍可解析

### 6.2 需要进入回滚验证链的回滚

满足以下任一情况时，允许回滚，但必须进入后续回滚验证链：

- 目标版本仅在测试环境验证过
- 当前版本相对目标版本发生过 `module_version` 或 `asset_version` 大幅变动
- 目标版本依赖 `field_aliases`、`id_redirects` 才能恢复当前存档

至于回滚验证链最少覆盖哪些路径、应沉淀哪些证据、什么情况下可以最终放行，统一转交 [附录-发布验证与放行闭环规范](附录-发布验证与放行闭环规范.md)。

### 6.3 默认禁止回滚

以下情况默认禁止回滚到稳定环境：

- 目标发布包无法读取当前存档版本
- 目标发布包无法识别当前存档依赖的关键模块版本或稳定 ID
- 当前游戏构建版本低于目标发布包要求
- 需要手工猜测字段映射或 ID 映射才能恢复

## 七、后台、游戏、测试三方职责分工

| 参与方 | 必须依赖的版本字段 | 不应承担的职责 |
| --- | --- | --- |
| 后台发布器 | `game_version`、`config_version`、`release_version`、`config_schema_version`、`module_versions`、`asset_versions`、`save_schema_version` | 不应把兼容结论只写在人工备注里 |
| 游戏读取器 | `required_game_version`、`config_schema_version`、`current_save_version`、`min_supported_save_version`、`blocked_save_versions`、存档元数据 | 不应自行发明第二套版本映射口径 |
| 测试与发布验证 | `game_version`、`config_version`、`module_versions`、`asset_versions`、`save_schema_version`、`validation_status` | 不应只验证“能启动”就宣布矩阵通过 |

统一约束：

1. 版本矩阵的业务汇总入口在发布记录。
2. 运行时兼容判断入口在 `manifest.yaml` + `compatibility.yaml`。
3. 测试报告必须写清“验证的是哪一组版本组合”，不能只写“当前版本”。

## 九、结论

- 本文只维护版本对象、版本组合和兼容/回滚判定口径。
- 发布包字段正文、记录载体字段和最终放行证据继续转交相邻发布专题，不在本文重复展开 `manifest` / `compatibility` 的字段表或落盘结构。

## 本文不负责的事项

- 不负责 `manifest.yaml`、`compatibility.yaml`、索引文件的字段清单本体。
- 不负责 `summary.yaml`、`matrix.yaml`、`rollback.yaml` 和存档元数据的最小落盘结构。
- 不负责发布前检查、发布后验证和最终放行的证据清单。
