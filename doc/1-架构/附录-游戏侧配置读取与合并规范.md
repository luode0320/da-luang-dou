# 游戏侧配置读取与合并规范 v1

## 目标

本规范定义游戏运行时如何读取后台发布包、如何合并全局与模块配置、如何缓存配置、如何处理兼容和热重载。目标是让游戏侧只负责“读、合、用”，不承担内容编辑职责。

## 1. 读取原则

1. 游戏只读取后台发布后的配置包。
2. 游戏不直接读取后台草稿。
3. 游戏不依赖后台页面结构，只依赖发布后的标准化结果。
4. 所有运行时配置都应有默认值兜底。
5. 配置读取失败时应明确报错，不允许静默失败。

## 2. 读取层级

运行时配置生效顺序统一为：

`实体覆盖 > 模块覆盖 > 全局默认 > 内置默认值`

其中：

- `实体覆盖`：某个实体目录自己的配置
- `模块覆盖`：模块内对全局内容的覆盖
- `全局默认`：公共默认配置
- `内置默认值`：代码兜底

## 3. 读取时机

建议在以下时机消费已定位好的正式配置结果：

- 游戏启动
- 进入模块前
- 进入关卡前
- 打开后台调试入口时
- 触发配置重载时

不建议在以下时机频繁全量读取：

- 每帧
- 每次攻击
- 每次掉落
- 每次碰撞

## 4. 缓存策略

### 4.1 缓存内容

建议缓存：

- 已发布配置索引
- 当前模块配置
- 当前关卡配置
- 当前角色配置
- 当前怪物配置
- 当前地图配置
- 当前技能配置

### 4.2 缓存规则

- 启动时加载索引
- 进模块时加载模块相关配置
- 进关卡时加载当前关卡所需配置
- 退出模块时释放模块缓存

## 5. 合并规则

### 5.1 标量字段

标量字段直接覆盖。

### 5.2 字典字段

字典字段按 key 合并。

### 5.3 列表字段

列表字段默认整表替换，除非明确标记支持追加或按 ID 合并。

### 5.4 素材字段

素材字段以发布结果为准，不允许运行时随意拼接不明路径。

## 6. 默认值策略

游戏侧必须为每类配置保留默认值，避免缺字段导致运行崩溃。

默认值优先级：

- 实体配置值
- 模块覆盖值
- 全局默认值
- 代码内置默认值

## 7. 兼容策略

### 7.1 新增字段

新增字段必须有默认值。

### 7.2 删除字段

删除字段必须保留兼容映射或迁移处理。

### 7.3 更名字段

更名字段必须保留旧别名读取。

### 7.4 废弃字段

废弃字段在兼容期内仍可读取，但不得再作为新逻辑主入口。

## 8. 热重载策略

可以考虑热重载的内容：

- 文案
- 音量
- 掉落表
- 部分数值参数
- 设置项

不建议热重载的内容：

- 关卡结构
- 存档 schema
- 核心战斗逻辑
- 实体类型定义

## 9. 错误处理与降级

当配置缺失或引用错误时，游戏侧应：

- 明确输出错误信息
- 标出缺失字段
- 标出问题实体 ID
- 标出问题来源目录
- 尽量进入降级模式而不是直接崩溃

降级只允许发生在运行时允许降级的字段或资源上；哪些情况属于通过、警告或阻断，继续转交版本矩阵与发布验证专题。

## 10. 调试输出

游戏侧调试输出建议包含：

- 当前发布版本
- 当前模块 ID
- 当前关卡 ID
- 当前实体 ID
- 实际生效的覆盖路径
- 缺失项和警告项

## 11. 运行时使用原则

游戏运行时只承担：

- 读取发布包
- 合并配置
- 缓存配置
- 执行配置
- 输出错误
- 记录追踪

不承担：

- 内容编辑
- 目录管理
- 素材上传
- 配置发布
- 配置审批

## 12. 结论

游戏侧配置读取的核心不是“直接加载一堆 YAML”，而是：

`消费正式配置入口 -> 定位正文 -> 合并层级 -> 应用默认值 -> 缓存运行 -> 记录追踪`

只要这条链路稳定，后台就可以持续扩展，而游戏侧不用因为新增模块而频繁改代码。
