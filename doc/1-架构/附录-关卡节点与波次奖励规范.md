# 关卡节点与波次奖励规范

## 目标

本附录用于把阶段 6 中已经提出、但还没有在长期架构层写硬的“关卡节点、单关配置、刷怪波次、Boss 节点、奖励配置和发布前校验”统一成长期契约。

它重点回答以下问题：

- `levels/`、`waves.yaml`、`rewards.yaml` 和掉落/刷怪表在长期结构上如何分层
- 单关配置最少应有哪些字段
- 刷怪波次和奖励配置最少应有哪些字段
- 默认 20 关、默认 60 秒和 Boss 节点如何写成统一约束
- 难度映射、地图引用、怪物池引用和奖励引用如何共用稳定 ID 口径
- 发布前最少要校验哪些断链、缺失和映射问题

## 一、分层总原则

### 1. 关卡节点层只负责单关、波次、奖励和节点关系契约，不替代主流程正文

- 关卡节点层负责：关卡编号、时长、节点关系、刷怪波次、Boss 触发、奖励结算和后续节点引用。
- 地图正文负责：地图素材、区域、刷怪区、事件区和碰撞定义。
- 怪物正文负责：怪物技能、阶段、掉落表和动作表现。
- 玩家如何从首页进入模块、如何在失败/退出后回到合法继续入口，继续转交 [6-业务链路-模块闯关与进度推进](6-业务链路-模块闯关与进度推进.md) 与 [附录-存档与资产生命周期规范](附录-存档与资产生命周期规范.md)。

### 2. 单关配置、波次配置和奖励配置必须拆层

- `config.yaml` 负责单关主配置。
- `waves.yaml` 负责刷怪波次和时间节奏。
- `rewards.yaml` 负责金币、道具池和通关奖励。
- 不建议把波次、奖励和关卡入口全部平铺进一个大文件。

### 3. 关卡参数长期默认写死为“20 关 + 60 秒”，节点关系由配置表达

- 每个模块长期默认 20 关。
- 每关长期默认 60 秒。
- 节点关系通过 `unlock_next_node_ids` 等字段表达，但“进入下一节点的运行时顺序”继续留在单链路正文中维护。
- 模块或关卡级覆盖只能调整具体数值和节点关系，不能破坏“默认有完整可校验主链路”这一长期假设。

## 二、目录与载体

长期建议至少采用以下结构：

```text
game/data/modules/<module_id>/
├── levels/<level_id>/
│   ├── config.yaml
│   ├── waves.yaml
│   ├── rewards.yaml
│   └── overrides/
├── spawns/<spawn_table_id>/
│   └── config.yaml
└── drops/<drop_table_id>/
    └── config.yaml
```

说明：

- `levels/<level_id>/` 负责单关配置。
- `spawns/<spawn_table_id>/` 负责可被多个关卡复用的刷怪表。
- `drops/<drop_table_id>/` 负责可被多个关卡或 Boss 复用的奖励与掉落表。
- 若某关只做少量差异化，优先通过 `overrides/` 或引用表覆盖，而不是复制整份波次和奖励配置。

## 三、单关主配置最小契约

### 3.1 `config.yaml` 最小字段

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `id` | 关卡稳定 ID |
| `module_id` | 所属模块 ID |
| `level_index` | 关卡编号 |
| `node_type` | 节点类型，如普通战斗、精英、Boss、事件 |
| `duration_sec` | 单关持续时间 |
| `map_id` | 地图引用 |
| `enemy_wave_pack_id` | 波次引用 |
| `boss_id` | Boss 引用 |
| `reward_pack_id` | 奖励引用 |
| `unlock_next_node_ids` | 后续解锁节点列表 |
| `difficulty_pack_id` 或 `difficulty` | 当前难度映射口径 |

### 3.2 主配置字段约束

- `id`、`module_id`、`level_index`、`node_type`、`map_id`、`enemy_wave_pack_id`、`reward_pack_id` 必填。
- `duration_sec` 缺失时默认 60。
- `node_type=Boss` 时，`boss_id` 必填。
- `unlock_next_node_ids` 可以为空，但若为空则该节点应能被明确识别为模块终点或分支终点。
- `level_index` 在同一模块下不得重复。

## 四、刷怪波次配置最小契约

### 4.1 `waves.yaml` 最小字段

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `wave_pack_id` | 当前波次配置 ID |
| `module_id` | 所属模块 |
| `level_id` | 所属关卡 |
| `waves` | 波次列表 |

### 4.2 单个 wave 最小字段

建议每个波次至少包含：

| 字段 | 作用 |
| --- | --- |
| `wave_id` | 波次稳定 ID |
| `start_sec` | 开始时间点 |
| `spawn_table_id` | 刷怪表引用 |
| `spawn_zone_id` | 地图刷怪区域 |
| `enemy_refs` | 怪物引用列表 |
| `spawn_count` | 总数量 |
| `spawn_interval_sec` | 刷怪间隔 |
| `clear_required` | 是否要求清完才能进入后续节点或阶段 |

### 4.3 波次约束

- `start_sec` 必须位于 `[0, duration_sec]` 区间。
- `spawn_table_id`、`spawn_zone_id`、`enemy_refs` 至少要有一条能解释“怎么刷怪”的合法路径。
- `enemy_refs` 中的怪物 ID 必须来自当前模块或全局已发布实体。
- `clear_required=true` 的波次不应出现在无法被清完的无限刷怪配置里。

## 五、奖励配置最小契约

### 5.1 `rewards.yaml` 最小字段

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `reward_pack_id` | 奖励配置 ID |
| `module_id` | 所属模块 |
| `level_id` | 所属关卡 |
| `coin_reward` | 基础金币奖励 |
| `drop_table_id` | 道具掉落表引用 |
| `clear_reward_refs` | 通关奖励引用 |
| `permanent_item_probability` | 永久金色道具概率 |

### 5.2 奖励约束

- `drop_table_id` 必填，哪怕只引用一个空表或基础表。
- `permanent_item_probability` 只允许作用于长期资产奖励，不得混到局内临时奖励里。
- Boss 节点默认允许挂载专属奖励表，但仍应复用同一字段口径，不额外造一套字段名。

## 六、刷怪表与掉落表最小契约

### 6.1 刷怪表 `spawns/<spawn_table_id>/config.yaml`

建议最少包含：

| 字段 | 作用 |
| --- | --- |
| `spawn_table_id` | 刷怪表稳定 ID |
| `enemy_pool_refs` | 怪物池 |
| `spawn_rule` | 刷怪规则 |
| `spawn_positions` | 刷怪位置或区域 |
| `difficulty_override_map` | 难度覆盖映射 |

### 6.2 掉落表 `drops/<drop_table_id>/config.yaml`

建议最少包含：

| 字段 | 作用 |
| --- | --- |
| `drop_table_id` | 掉落表稳定 ID |
| `coin_rule` | 金币掉落规则 |
| `temporary_item_pool_refs` | 临时道具池 |
| `permanent_item_pool_refs` | 永久道具池 |
| `boss_bonus_pool_refs` | Boss 额外奖励池 |
| `difficulty_override_map` | 难度覆盖映射 |

## 七、节点关系与默认结构规则

### 7.1 默认结构

长期默认主链路至少满足：

1. 模块存在 20 个默认主线节点
2. 关卡编号从 `1` 连续到 `20`
3. 主线节点之间存在可解析的后续节点关系
4. 模块终点明确可识别

### 7.2 允许的节点类型

建议最少支持：

- `normal`
- `elite`
- `boss`
- `event`

### 7.3 节点关系约束

- `unlock_next_node_ids` 不允许引用不存在节点。
- 主线节点默认应保证从起点可达终点。
- 若存在分支节点，仍要保证至少一条可验证的主链路。
- 本附录只要求节点关系“可校验、可解析”；真正的进入顺序、结算后跳转和恢复判断继续由业务链路正文负责。

## 八、Boss 节点与难度映射规则

### 8.1 Boss 节点

- Boss 节点必须显式标识 `node_type=Boss`。
- Boss 节点必须绑定合法 `boss_id`。
- Boss 节点默认应支持引用 Boss 阶段、狂暴概率和专属奖励表，但这些细节通过怪物/Boss 配置和奖励表承接，不直接散写在节点主配置里。

### 8.2 难度映射

- 关卡层只负责引用难度映射入口。
- 具体怪物技能开放、Boss 形态开放、狂暴概率和掉落倍率，优先由难度包或怪物/Boss 配置承接。
- 关卡层不得自己发明与难度包冲突的第二套难度字段。

## 九、发布前最小校验集合

长期建议发布前至少校验以下内容：

| 检查项 | 最小要求 | 阻断条件 |
| --- | --- | --- |
| 20 关连续性 | 默认主线存在 20 个连续节点 | 缺关、重号、断链 |
| 关卡时长 | 默认值可回落到 60 秒 | 时长非法或缺失且无默认 |
| 地图引用 | 每关都有合法 `map_id` | 地图不存在 |
| 波次引用 | 每关都有合法 `enemy_wave_pack_id` | 波次文件缺失或引用失效 |
| 奖励引用 | 每关都有合法 `reward_pack_id` | 奖励表缺失 |
| Boss 节点合法性 | Boss 关有合法 `boss_id` | Boss 节点无 Boss 引用 |
| 难度映射 | 难度入口可解析 | 难度引用缺失或冲突 |
| 后续节点关系 | `unlock_next_node_ids` 可解析 | 后续节点不存在 |

## 十、运行时最小输出字段

关卡开始、推进与结算时，运行时最少应输出：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 当前模块 |
| `level_id` | 当前关卡 |
| `level_index` | 当前关卡编号 |
| `node_type` | 当前节点类型 |
| `map_id` | 当前地图 |
| `enemy_wave_pack_id` | 当前波次配置 |
| `reward_pack_id` | 当前奖励配置 |
| `duration_sec` | 当前时长 |
| `result` | 当前结果 |
| `unlock_next_node_ids` | 当前节点关系输出 |

## 十二、结论

关卡配置长期最容易失控的，不是“某一关刷了什么怪”，而是下面三层边界有没有先写硬：

- 单关主配置负责什么
- 波次配置负责什么
- 奖励配置负责什么

只要这三层先写硬，再把默认结构、60 秒默认值、Boss 节点和难度映射校验写进长期规则，后续阶段 6 的实现、验证和发布校验就能围绕同一套契约推进；而“玩家如何连续推进、失败后如何恢复、开始新游戏如何重置”继续只留在业务链路和生命周期专题。
