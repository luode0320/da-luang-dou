## 实施总原则

1. 先固化配置真源、发布包契约、读取规则和兼容边界，再进入 Godot 与后台骨架实现。
2. 每个阶段同时覆盖游戏侧目标与后台配置目标，不把后台当成事后补充。
3. 正式素材先走 `2d-asset-design` / `imagegen` 设计预览，再进入 Godot 资源接入。
4. 调试验证、发布回滚和 Windows 打包不是附加项，而是阶段 7 必须收口的正式能力。

## 当前优先闭环

### 第一优先级

- 阶段 0 配置驱动基础设施
- 发布包最小契约
- 运行时读取与错误处理
- 版本矩阵与兼容判定

### 第二优先级

- 模块选择与启动入口
- 地图、角色、怪物、关卡四条主内容链路
- 自动存档与节点推进

### 第三优先级

- 配置冒烟
- 调试直达
- Windows 打包安装
- 发布放行

## 当前推荐推进顺序

1. 固定 `game/data/` 目录、发布包载体和读取顺序。
2. 建立后台最小闭环，只先支持草稿、校验、预览和发布。
3. 初始化 Godot 4 最小工程，读取已发布配置进入首页和模块入口。
4. 打通模块入口、角色选择、关卡战斗和自动存档最小闭环。
5. 扩展地图契约、角色契约、怪物契约和关卡契约对应的内容配置。
6. 补齐调试直达、回滚验证、配置冒烟和 Windows 打包收口。

## 关键阻断与依赖

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| 阶段 0 未闭环 | 后续所有阶段都会漂移到硬编码实现。 | 先以架构附录为基线完成发布包与读取规则。 |
| Godot 工程未建立 | 地图、角色、怪物和关卡无法验证运行链路。 | 在进入运行态实施前确认 Godot 工程可启动。 |
| 后台最小闭环缺失 | 游戏侧无法稳定消费配置真源。 | 先做最小草稿、校验、预览和发布能力。 |
| 素材流程过早跳到接入 | 容易把占位图误当成最终资产。 | 先设计预览，再接入 Godot。 |
| 调试验证收口滞后 | 阶段 7 会堆积大量不可见风险。 | 在阶段 0 就预埋版本、日志和调试入口口径。 |
