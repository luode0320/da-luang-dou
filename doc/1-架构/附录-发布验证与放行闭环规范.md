## 一、闭环总原则

### 1. 验证闭环分三段

长期发布闭环默认固定分三段，不允许只做其中一段就宣称“可上线”：

1. 发布前检查
2. 发布后验证
3. 最终放行

### 2. 放行结论依赖证据，不依赖口头说明

- “能启动一次”不等于验证完成。
- “发布记录已生成”不等于可放行。
- “后台显示通过”不等于游戏恢复、回滚和打包都通过。
- 所有放行结论必须落在可复核证据上，而不是人工备注。

### 3. 发布、回滚、恢复使用同一套版本口径

- 发布验证、回滚验证、存档恢复验证都必须引用同一组 `game_version`、`config_version`、`release_version`、`module_version`、`asset_version`、`save_schema_version`。
- 不允许测试报告、验收结论和后台发布记录各自维护不同版本解释。

## 二、三阶段闭环

### 2.1 发布前检查

目标：

- 证明“这组草稿可以组成一份有效发布包”

默认最小检查项：

| 检查项 | 最小要求 | 阻断条件 |
| --- | --- | --- |
| 发布包契约完整性 | 关键发布契约文件与最小索引集合可生成 | 任一关键发布契约文件缺失 |
| 版本矩阵完整性 | 发布记录、版本矩阵、兼容字段齐全 | 关键版本字段缺失或互相矛盾 |
| 模块完整性 | 模块入口、关卡连续性、Boss 节点、难度映射可解析 | 模块入口缺失、关卡断链 |
| 引用完整性 | 实体引用、素材引用、稳定 ID 可定位 | 引用断链、素材缺失、稳定 ID 冲突 |
| 存档兼容性 | 兼容契约可解释当前支持范围 | 兼容映射缺失且存在破坏性变更 |
| 回滚可行性 | 存在明确可回滚目标与验证路径 | 没有目标版本或无法判断兼容 |
| 打包参数完整性 | Windows 构建渠道、版本号、安装元信息齐全 | 打包参数缺失，无法形成构建产物 |

### 2.2 发布后验证

目标：

- 证明“已经生成的发布结果可被真实运行、恢复和回滚”

默认最小验证项：

| 验证项 | 最小要求 | 阻断条件 |
| --- | --- | --- |
| 配置冒烟启动 | 首页、模块、地图、角色、怪物、关卡可通过发布包读取 | 启动后主链路任一点无法读取 |
| 关卡直达调试 | 可指定模块、关卡、难度、角色、道具、存档状态启动 | 调试入口无法覆盖问题定位需要 |
| 存档兼容恢复 | 兼容范围内旧自动存档可恢复 | 旧存档应兼容却无法恢复 |
| 回滚验证 | 可回到目标版本并完成最小冒烟 | 回滚后无法进入主链路或恢复当前模块 |
| Windows 打包验证 | exe、安装器、快捷方式链路可执行 | 无法安装或启动到首页 |
| 运行追踪 | 能输出版本、模块、关卡、实体、错误和警告 | 无法定位当前运行版本或关键失败点 |

### 2.3 最终放行

目标：

- 证明“当前版本不是仅能验证，而是具备进入目标环境的最小交付条件”

最终放行默认必须同时满足：

1. 发布前检查已通过
2. 发布后验证已通过
3. 已形成发布记录、回滚记录和验证证据
4. 已形成最终验收结论
5. 不存在未处理阻断项

## 三、最小验证集合

### 3.1 配置冒烟验证最小集合

配置冒烟不是全量回归，但至少应覆盖一条完整读取链：

1. 启动游戏并读取当前发布版本入口
2. 进入首页并加载首页预览配置
3. 进入模块选择并解析模块索引
4. 进入目标模块地图并读取模块正文配置
5. 进入关卡并加载角色、怪物、地图与素材引用
6. 输出当前版本标识与当前场景定位标识

若以上任何一步失败，默认不能宣称“发布包可运行”。

### 3.2 关卡直达调试最小边界

关卡直达是验证和排障入口，不是正式玩家入口。

调试入口最少应支持指定：

- 目标模块定位
- 目标关卡或节点定位
- 当前难度定位
- 当前角色定位
- 外部永久道具搭配或等价验证载荷
- 存档状态或恢复模式

统一约束：

- 关卡直达只能服务开发、测试和回滚验证。
- 关卡直达必须明确标识当前使用的是哪组发布版本。
- 关卡直达不可绕开发布契约态直接读取后台草稿。

### 3.3 存档兼容验证最小路径

最小验证路径至少包含：

1. 准备一份兼容范围内的历史自动存档
2. 使用当前发布包读取兼容契约
3. 校验 `save_schema_version`、`last_release_version`、`module_version`
4. 恢复到当前模块合法继续入口
5. 继续进入一场关卡或恢复后界面
6. 记录成功、警告或阻断结论

若恢复成功但存在字段降级、ID 重定向或素材替换，应记录为“通过并带警告”，不能直接写成“完全无风险”。

### 3.4 回滚验证最小路径

最小回滚验证至少包含：

1. 选择目标 `to_release_version`
2. 确认当前 `game_version` 满足目标版本要求
3. 执行回滚并生成本次回滚记录
4. 重新启动游戏读取目标发布包
5. 完成一次配置冒烟启动
6. 读取当前活跃模块自动存档
7. 记录回滚验证摘要

### 3.5 Windows 打包与安装验证最小路径

最小路径至少包含：

1. 生成 Windows exe
2. 生成安装器或标准安装包
3. 安装后创建快捷方式
4. 通过安装产物启动到首页
5. 首页可读取当前发布配置并显示关键版本

若只能本地裸运行 exe，但安装流程未验证，默认只能记为“验证通过待放行”。

## 四、四方证据职责

### 4.1 后台侧必须输出的证据

至少包括：

- 发布记录摘要
- 校验记录
- 版本矩阵记录
- 审计记录
- 回滚记录（如发生回滚）

这些证据的具体落盘文件名与最小字段，统一以 [附录-发布记录回滚记录与存档元数据契约](附录-发布记录回滚记录与存档元数据契约.md) 为准。

### 4.2 游戏侧必须输出的证据

至少包括：

- 当前版本标识
- 当前模块与关卡定位标识
- 关键读取路径与阻断日志
- 存档恢复结果
- 回滚后读取结果

### 4.3 测试侧必须输出的证据

至少包括：

- 本轮验证对应的版本组合
- 冒烟验证范围
- 存档兼容验证结果
- 回滚验证结果
- Windows 打包验证结果

### 4.4 验收侧必须输出的证据

至少包括：

- 当前验收依据的需求与验收标准
- 本轮验证是否覆盖最小放行集合
- 剩余风险和未覆盖项
- 结论是“最终放行”“验证通过待放行”还是“阻断”

## 五、放行结论分级

### 5.1 最终放行

满足以下条件时才可判定：

- 发布前检查通过
- 发布后验证通过
- 兼容范围内旧存档可恢复
- 回滚验证通过或明确无需回滚验证且有合理依据
- Windows 打包与安装流程通过
- 审计、测试、验收证据完整

### 5.2 验证通过待放行

满足以下条件时可判定：

- 主链路和版本契约验证通过
- 仍有少量非阻断项未完成
- 这些缺口不会影响当前主链路可验证性，但会影响正式交付信心

典型场景：

- 裸 exe 验证通过，但安装器流程未做
- 主链路兼容验证通过，但回滚只完成静态判定未做实跑
- 测试证据齐全，但最终验收文档尚未落盘

### 5.3 必须阻断

出现以下任一情况默认阻断：

- 关键版本字段缺失或矛盾
- 发布包关键文件缺失
- 主链路冒烟失败
- 兼容范围内历史存档无法恢复
- 回滚目标不可用或回滚后主链路失败
- Windows 安装产物无法启动到首页
- 无法提供当前版本和验证结果对应关系

## 六、推荐验证顺序

长期推荐按以下顺序组织发布验证，避免先做昂贵验证再被前置问题打回：

1. 发布前结构与版本检查
2. 配置冒烟启动
3. 关卡直达验证
4. 存档兼容验证
5. 回滚验证
6. Windows 打包与安装验证
7. 验收收口与最终放行
