# 关卡直达调试与验证入口规范

## 目标

本附录用于把阶段 7 中已经提出、但还没有在长期架构层写硬的“关卡直达调试入口、发布后验证入口、配置冒烟测试清单和验证输出字段”统一成长期契约。

它重点回答以下问题：

- `game/data/control/debug/` 下最少应该有哪些配置载体
- `launch_profiles.yaml` 最少应该定义哪些字段
- `smoke_tests.yaml` 最少应该定义哪些字段
- 正式玩家入口与调试直达入口如何隔离
- 调试直达、存档恢复、发布后验证和回滚验证如何共用同一套入口参数口径
- 运行时、测试和验收最少要输出哪些验证结果

## 一、分层总原则

### 1. 调试入口属于控制配置层，不属于正式玩家默认入口

- 调试直达、发布后验证和回滚验证都属于开发测试验证入口。
- 这些入口可以绕过首页和模块选择的正常玩家交互流程，但不能绕过发布契约态。
- 不允许通过调试入口直接读取后台草稿、未发布模块或临时页面状态。

### 2. 调试入口与正式玩家入口共用版本口径，不共用展示语义

- 正式玩家入口和调试入口都必须明确当前 `release_version`、`game_version` 和目标模块。
- 但正式玩家入口只暴露“继续上次”“开始新游戏”“模块选择”等正常语义。
- 调试入口可以暴露“指定模块”“指定节点”“指定难度”“指定恢复策略”等技术语义。

### 3. 调试入口配置只表达“怎么验证”，不表达正文玩法配置

- `launch_profiles.yaml` 负责定义验证入口如何启动。
- `smoke_tests.yaml` 负责定义验证任务如何批量执行。
- 角色、怪物、关卡、地图和掉落的正文配置仍然由发布包正文承接。

## 二、目录与载体

长期建议在 `game/data/control/debug/` 下至少维护：

```text
game/data/control/debug/
├── launch_profiles.yaml   # 调试直达与验证入口配置
└── smoke_tests.yaml       # 冒烟与最小验证任务清单
```

说明：

- 这两个文件都属于控制配置层长期契约，不进入模块正文目录。
- 运行时只应读取已发布版本中的调试配置，不直接拼接后台草稿。
- 若某个发布版本不允许调试入口生效，也应显式通过配置开关或构建渠道控制，而不是删除字段绕过契约。

## 三、入口类型

长期建议把验证入口至少拆成以下四类：

| 入口类型 | 用途 | 典型场景 |
| --- | --- | --- |
| `debug_direct` | 开发期问题定位与关卡直达 | 指定模块、关卡、角色、难度快速复现 |
| `smoke_entry` | 配置冒烟与主链路抽检 | 首页、模块、关卡最小读取链验证 |
| `save_resume_verify` | 存档恢复与兼容验证 | 指定历史自动存档恢复 |
| `rollback_verify` | 回滚后的重新启动与恢复验证 | 切到目标 `release_version` 后执行最小冒烟 |

统一约束：

- `debug_direct` 不作为正式玩家默认入口。
- `smoke_entry` 不承担玩法入口职责，只承担验证职责。
- `save_resume_verify` 必须走合法存档恢复链，不能伪造恢复结果。
- `rollback_verify` 必须绑定明确目标版本，不允许只写“上一版”这类模糊描述。

## 四、`launch_profiles.yaml` 最小契约

### 4.1 顶层建议字段

建议最少包含：

| 字段 | 作用 |
| --- | --- |
| `profiles` | 启动配置列表 |
| `default_profile_id` | 默认调试配置 ID |
| `enabled_build_channels` | 允许生效的构建渠道 |
| `generated_from_release_version` | 当前配置对应的发布版本 |

### 4.2 单个 profile 最小字段

建议每个 profile 至少包含：

| 字段 | 作用 |
| --- | --- |
| `profile_id` | 启动配置稳定 ID |
| `entry_type` | 入口类型，如 `debug_direct`、`smoke_entry`、`save_resume_verify`、`rollback_verify` |
| `release_version` | 当前目标发布版本 |
| `game_version` | 当前目标游戏构建版本 |
| `module_id` | 目标模块 ID |
| `node_id` 或 `level_id` | 目标节点或关卡 |
| `difficulty_id` | 目标难度 |
| `role_id` | 目标角色 |
| `external_item_ids` | 外部永久道具列表 |
| `save_resume_mode` | 恢复模式，如 `none`、`use_current_save`、`use_fixture_save` |
| `fixture_save_id` | 指定测试存档 ID |
| `start_phase` | 启动阶段，如首页、模块地图、关卡前或关卡内 |
| `purpose` | 启动用途说明 |
| `expected_result` | 最小预期结果摘要 |
| `enabled` | 当前 profile 是否启用 |

### 4.3 字段约束

- `release_version` 必填；调试入口不得缺少发布版本。
- `module_id` 必填；不能存在无模块归属的关卡直达入口。
- `node_id` 与 `level_id` 至少二选一；若两者都缺失，则只能是首页级冒烟入口。
- `save_resume_mode=use_fixture_save` 时，`fixture_save_id` 必填。
- `entry_type=rollback_verify` 时，`purpose` 中必须能定位回滚验证目标。
- `enabled=false` 的 profile 可以保留为模板，但运行时默认不得直接执行。

## 五、`smoke_tests.yaml` 最小契约

### 5.1 顶层建议字段

建议最少包含：

| 字段 | 作用 |
| --- | --- |
| `test_suite_id` | 当前冒烟测试清单 ID |
| `release_version` | 本轮清单对应的发布版本 |
| `tests` | 测试项列表 |

### 5.2 单个 test 最小字段

建议每个测试项至少包含：

| 字段 | 作用 |
| --- | --- |
| `test_id` | 测试项稳定 ID |
| `entry_profile_id` | 引用的启动配置 ID |
| `test_type` | 测试类型，如 `home_smoke`、`module_smoke`、`level_smoke`、`save_resume_smoke`、`rollback_smoke` |
| `required` | 是否为放行前必测项 |
| `expected_release_version` | 预期版本 |
| `expected_module_id` | 预期模块 |
| `expected_level_id` | 预期关卡 |
| `expected_result` | 最小成功标准 |
| `blocking_on_fail` | 失败后是否直接阻断 |

### 5.3 建议的最小测试集合

长期建议至少有以下 5 类必测项：

1. 首页读取冒烟
2. 模块索引读取冒烟
3. 指定模块关卡直达冒烟
4. 历史存档恢复冒烟
5. 回滚后重新启动冒烟

## 六、正式玩家入口与调试入口隔离规则

### 6.1 入口展示隔离

- 正式玩家首页、模块选择页和开始/继续入口不直接展示 `profile_id`、`fixture_save_id`、`rollback` 等技术字段。
- 调试入口可显示这些字段，但默认只在开发或测试构建渠道开放。

### 6.2 数据来源隔离

- 正式玩家入口读取控制配置、模块索引、模块正文和合法自动存档。
- 调试入口在此基础上额外读取 `game/data/control/debug/` 下的调试配置。
- 无论哪种入口，都必须读取发布后的版本化配置，不读取后台草稿。

### 6.3 能力边界隔离

- 正式玩家入口不能伪造外部道具、存档状态或任意关卡跳转。
- 调试入口允许注入这些验证参数，但必须在日志里显式记录。
- 调试入口的存在不等于正式玩家一定能访问该关卡；它只用于验证。

## 七、恢复与验证参数统一口径

长期建议把以下参数作为调试、恢复和验证共用口径：

| 参数 | 用途 |
| --- | --- |
| `release_version` | 绑定验证目标版本 |
| `game_version` | 绑定目标构建版本 |
| `module_id` | 绑定目标模块 |
| `node_id` / `level_id` | 绑定目标节点或关卡 |
| `difficulty_id` | 绑定目标难度 |
| `role_id` | 绑定目标角色 |
| `external_item_ids` | 绑定外部永久道具 |
| `save_resume_mode` | 绑定恢复方式 |
| `fixture_save_id` | 绑定测试存档 |
| `entry_type` | 标识入口类型 |

统一约束：

- 测试报告、验收说明、运行日志和回滚记录都应复用这组参数口径。
- 不允许运行日志里只写关卡名、测试报告里只写模块名，而没有统一稳定 ID。

## 八、运行时最小输出字段

运行时在调试直达、恢复验证和回滚验证场景下，最少应输出：

| 字段 | 作用 |
| --- | --- |
| `entry_type` | 当前入口类型 |
| `profile_id` | 当前启动配置 ID |
| `release_version` | 当前发布版本 |
| `game_version` | 当前游戏版本 |
| `module_id` | 当前模块 |
| `node_id` / `level_id` | 当前节点或关卡 |
| `difficulty_id` | 当前难度 |
| `role_id` | 当前角色 |
| `save_resume_mode` | 当前恢复模式 |
| `resume_result` | 恢复结果 |
| `validation_status` | 当前验证状态 |
| `blocking_reason` | 当前阻断原因 |

说明：

- 即使验证失败，也要输出 `validation_status` 和 `blocking_reason`。
- 若为 `save_resume_verify`，`resume_result` 不能缺失。
- 若为 `rollback_verify`，日志里必须能看出当前实际已切换到哪个版本。

## 九、测试与验收最小输出字段

### 9.1 测试侧

测试验证最少应记录：

- `test_id`
- `entry_profile_id`
- `release_version`
- `module_id`
- `level_id`
- `result`
- `blocking_on_fail`
- `notes`

### 9.2 验收侧

验收最少应记录：

- 本轮使用的 `release_version`
- 覆盖的入口类型
- 是否完成首页、模块、关卡、恢复、回滚最小集合
- 是否存在阻断项
- 放行结论

## 十、异常与阻断边界

| 异常场景 | 处理方式 | 最终状态 |
| --- | --- | --- |
| 调试入口缺少 `release_version` | 视为验证证据不足 | 阻断验证 |
| 调试入口直接读取后台草稿 | 视为违规实现 | 阻断上线 |
| `smoke_tests.yaml` 引用不存在的 `entry_profile_id` | 视为测试契约失效 | 阻断验证 |
| `save_resume_verify` 缺少恢复结果 | 视为恢复验证无效 | 阻断放行 |
| 回滚验证无法标识目标版本 | 视为回滚证据无效 | 阻断放行 |
| 正式玩家入口暴露调试技术参数 | 视为入口隔离失败 | 回退修正 |

## 十二、结论

关卡直达调试与发布后验证长期最容易失控的，不是“能不能跳到某一关”，而是下面三件事有没有先写硬：

- 调试入口是否绑定正式发布版本
- 调试入口是否和正式玩家入口彻底分层
- 验证结果是否能输出为可复核证据

只要这三件事先写硬，后续调试直达、配置冒烟、存档恢复、回滚验证和验收放行就能围绕同一套长期入口契约推进。
