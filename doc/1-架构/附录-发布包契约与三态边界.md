# 发布包契约与三态边界

## 目标

本附录用于把当前架构里最容易混淆的三件事拆开写清楚：

- 后台正在编辑的草稿和预览数据是什么
- 游戏真正可以读取的发布包契约是什么
- 游戏运行时内存里的已解析配置又是什么

如果这三层边界不写硬，后续很容易出现“游戏直接读后台草稿”“发布包和运行态缓存混写”“目录骨架被误当成最终运行协议”的问题。

## 核心结论

长期架构上，配置生产与消费必须区分为三态：

1. 后台编辑态
2. 发布契约态
3. 游戏运行态

这三态的责任、读写权限和生命周期不同，不能混用。

## 一、后台编辑态

### 定义

后台编辑态是内容生产过程中的可变工作区，服务于编辑、绑定、预览、校验和待发布确认。

### 主要特征

- 允许频繁修改
- 允许存在草稿、不完整字段和待修复校验项
- 允许按后台页面结构或编辑器结构组织
- 可以附带预览态信息、临时备注和未发布版本标记

### 责任模块

- `admin/frontend/`
- `admin/backend/`

### 允许操作

- 创建草稿
- 编辑字段
- 绑定素材
- 执行校验
- 生成预览
- 发起发布
- 回滚到历史已发布版本作为新草稿基线

### 禁止操作

- 让游戏运行时直接读取草稿
- 把后台页面结构当成游戏运行协议
- 把未校验通过的配置直接落成运行输入

## 二、发布契约态

### 定义

发布契约态是后台发布后生成的、供游戏和打包流程消费的标准化只读配置包。它不是后台草稿，也不是游戏内存缓存，而是两者之间唯一可信的交付契约。

### 主要特征

- 只读
- 版本化
- 可校验
- 可回滚
- 可追踪
- 独立于后台页面结构

### 责任模块

- `admin/backend/` 负责生成
- `game/data/published/` 负责承载
- `tools/windows/` 可引用该层进行打包

### 最低契约要求

每个可运行发布版本在三态边界层至少应同时具备以下四类正式载体：

- 版本与来源入口
- 兼容入口
- 索引入口
- 正文配置入口

这些入口在目录、文件名和最小字段上的具体展开，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。

### 长期结构边界

本文在结构层只固定三件事：

1. 运行时唯一正式输入根目录长期收口到 `game/data/published/`
2. 每个发布版本都必须是独立、只读、可追踪的版本化目录
3. 版本目录内部至少要有“版本入口 + 兼容入口 + 索引入口 + 正文配置入口”四类载体

这些载体在目录、文件名和最小字段上的具体展开，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。

### 当前仓库与长期结构边界的关系

- 当前仓库已经在 `game/data/` 下建立了 `control/`、`global/`、`modules/`、`published/` 四类逻辑命名空间的骨架。
- 这些骨架当前主要用于长期目录规划和阶段 0 设计锚点，不应被理解为“游戏现在已经直接读取这些目录”。
- 真正的运行时输入，长期目标仍应收口到 `game/data/published/` 下的版本化发布包。

## 三、游戏运行态

### 定义

游戏运行态是 Godot 运行时在消费正式发布结果后形成的内存态结果。

### 主要特征

- 只在运行时存在
- 只作为正式发布结果的消费结果存在
- 生命周期受当前游戏进程与会话控制
- 不应回写发布包正文

### 责任模块

- `game/`

### 允许操作

- 识别目标发布版本
- 消费正式发布结果
- 形成运行时可用的内存态结果
- 输出运行时诊断结果

### 禁止操作

- 直接编辑发布包正文
- 依赖后台草稿目录
- 把运行时缓存当成长期真源

## 四、三态之间的单向关系

正确的数据流应保持单向：

`后台编辑态 -> 发布契约态 -> 游戏运行态`

### 允许的回流

允许的回流只有“以已发布版本为基线重新生成新草稿”这种后台工作流回流，而不是游戏运行时反向写回发布包。

### 禁止的逆向路径

- `游戏运行态 -> 发布契约态`
- `游戏运行态 -> 后台编辑态`
- `后台编辑态 -> 游戏运行态`

## 五、对目录设计的直接约束

### `admin/`

- 只负责编辑态和发布动作
- 不作为游戏直接读取目录

### `game/data/control/`、`game/data/global/`、`game/data/modules/`

- 当前阶段可作为逻辑命名空间、模板骨架和目录规划锚点
- 不能据此放松“游戏长期只读发布包”的约束

### `game/data/published/`

- 长期作为运行态输入的唯一根入口
- 后续实现时优先向版本化、可回滚、可追踪结构演进

## 六、对后台设计的直接约束

- 后台前端围绕“草稿编辑与预览”组织页面，不围绕游戏运行目录组织页面
- 后台后端围绕“校验、组包、版本、回滚、审计”组织服务，不把发布动作简化成文件复制
- 发布动作必须把草稿收敛为标准化只读发布结果，不只停留在零散草稿或页面结构

## 七、对游戏实现的直接约束

- Godot 运行时先识别目标发布版本，再消费正式发布结果
- 运行时只从发布契约态派生内存态结果，不自建并行长期真源
- 任何调试、热重载、兼容处理和异常定位都不得绕过发布契约态直接读取后台草稿

## 九、结论

- 本文只回答三态边界和单向数据流。
- 正式入口、最小字段、运行时消费细则、版本矩阵、记录载体和放行闭环分别由相邻发布专题承接，不在本文重复展开。

## 本文不负责的事项

- 不负责 `manifest.yaml`、`compatibility.yaml` 和索引文件的最小字段清单。
- 不负责运行时精确读取顺序、合并策略、缓存策略、热重载范围和错误输出字段。
- 不负责 `game_version`、`config_version`、`release_version` 等版本兼容判定规则。
- 不负责 `summary.yaml`、`rollback.yaml`、存档元数据和最终放行证据结构。
