# 目录树与模块职责边界去重收口

## 背景

- 当前长期架构主入口已经稳定为 `1-总架构.md`、`2-目录树.md`、`3-模块职责.md`、`4-主要业务链路.md`。
- 继续优化架构设计时，`2-目录树.md` 与 `3-模块职责.md` 仍存在明显重叠：
  - `2-目录树.md` 里写了过多“模块如何协作”的说明。
  - `3-模块职责.md` 里写了过多“目录里具体放哪些载体”的说明。
- 这会让两个主入口都变成半个“目录 + 模块 + 专题”的混合页，不利于后续继续收紧真相源。

## 本轮收口目标

- 让 `2-目录树.md` 只回答“路径长什么样、目录放什么、哪些内容不应该放进去”。
- 让 `3-模块职责.md` 只回答“模块谁负责什么、输入输出是什么、依赖方向怎么走、协作边界怎么划”。
- 把字段级和正文级问题继续压回相应附录，不在两个主入口重复展开。

## 本轮调整结果

### `2-目录树.md`

- 新增“本文负责什么 / 本文不负责什么”。
- 删除原先大段“控制层 / 全局层 / 模块层 / effects 层如何协作”的模块级解释。
- 保留：
  - 当前真实目录骨架
  - 目标发布包结构建议
  - 路径级边界
  - 目录职责表
- 收紧后的口径是：
  - `control/` 只管控制层路径
  - `global/` 只管公共默认路径
  - `modules/` 只管单模块路径
  - `published/` 只管发布产物路径

### `3-模块职责.md`

- 新增“本文负责什么 / 本文不负责什么”。
- 新增“依赖方向”章节，显式说明：
  - 后台前端 -> 后台后端 -> 发布契约层 -> 游戏运行层
  - 控制 / 全局 / 模块配置层先经过发布流程，再进入运行时
  - 进度与存档层不反向定义发布契约
- 收紧模块清单中三层配置模块的输出描述，不再把它们写成具体目录载体清单。
- 把具体载体字段、正文目录和名单文件名，统一改回“对应附录承接”，不在模块职责正文重复展开。

## 当前分工结论

| 文档 | 现在主要回答的问题 | 不再回答的问题 |
| --- | --- | --- |
| `2-目录树.md` | 当前有哪些路径、目标发布包结构如何组织、每条路径允许/禁止放什么 | 模块之间谁依赖谁、谁对什么结果负责 |
| `3-模块职责.md` | 各模块职责、输入输出、依赖方向、协作边界 | 某个目录里有哪些具体文件、某个载体放在哪条路径 |

## 后续维护规则

- 若新增目录、调整路径或修改某条路径允许内容，优先更新 `2-目录树.md`。
- 若新增模块、调整输入输出或改变依赖方向，优先更新 `3-模块职责.md`。
- 若变化已经下沉到 `home.yaml`、`module.yaml`、`manifest.yaml`、`role_roster.yaml`、`effect_ref` 这类字段或正文载体，优先更新对应附录，而不是回到主入口写第二遍。

## 结论

- 本轮不是新增更多架构专题，而是把已有主入口的边界重新钉死。
- 收口后：
  - `2-目录树.md` 更接近“路径级真相源”
  - `3-模块职责.md` 更接近“模块级真相源”
- 后续如果还出现重叠，默认继续按“路径问题回目录树，模块问题回模块职责，字段问题回附录”的规则收紧。
