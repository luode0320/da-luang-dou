# 模块正文配置与模块名单规范

## 目标

本附录用于把阶段 2 中已经提出、但尚未在长期架构层写硬的“模块正文配置、模块名单载体、模块角色 / 怪物 / 地图可见范围和模块索引边界”统一成长期规则。

它重点回答以下问题：

- `modules_index.yaml` 和 `modules/<module_id>/module.yaml` 的边界到底在哪里
- 模块正文应该承载哪些最小字段
- 模块角色、怪物、地图和关卡名单应该放在哪类载体里
- `role_roster.yaml` 或等价名单载体如何与模块正文协作
- 模块可见性、模块状态和模块发布边界如何长期稳定

## 一、分层总原则

### 1. 模块索引是入口层，模块正文是模块本体

- `modules_index.yaml` 负责“有哪些模块、模块入口显示什么、默认从哪里进入、是否可见”。
- `modules/<module_id>/module.yaml` 负责“这个模块本体是什么、主题是什么、默认地图 / 节点是什么、这个模块允许哪些名单载体参与运行时”。
- 不允许把模块正文的实体名单、规则正文和内部细节全部堆进索引层。

### 2. 模块正文不等于实体名单集合

- 模块正文负责模块级稳定元信息与默认入口行为。
- 角色名单、怪物名单、地图名单和其他可选实体集合，长期应由独立名单载体承载。
- 名单载体引用实体正文，不复制实体正文。

### 3. 模块名单只回答“本模块允许哪些实体参与”

- 名单载体负责模块内可见、可选、默认推荐、被禁用和按难度开放的实体集合。
- 实体本体属性、技能、动画、地图区域和关卡波次仍由各自实体正文承载。
- 名单载体不替代角色 / 怪物 / 地图 / 关卡的正文字段。

## 二、目录与载体分层

长期建议每个模块至少维护以下载体：

```text
game/data/modules/<module_id>/
├── module.yaml
├── role_roster.yaml
├── enemy_roster.yaml
├── map_roster.yaml
├── levels/
├── characters/
├── enemies/
├── bosses/
├── maps/
└── assets/
```

说明：

- `module.yaml` 是模块本体总配置。
- `role_roster.yaml`、`enemy_roster.yaml`、`map_roster.yaml` 是模块名单层。
- `characters/`、`enemies/`、`bosses/`、`maps/`、`levels/` 是模块正文或模块级覆盖层。

## 三、模块索引与模块正文边界

### 3.0 模块层三件套速查

| 载体 | 主要职责 | 不负责 |
| --- | --- | --- |
| `modules_index.yaml` | 模块入口显示、顺序、可见性、默认入口模式。 | 本文不展开；入口层字段由 [附录-模块索引与启动入口规范](附录-模块索引与启动入口规范.md) 负责。 |
| `modules/<module_id>/module.yaml` | 模块主题、默认地图、默认起点、默认推荐角色、名单引用、模块发布状态。 | 首页入口显示、公共导航、首页预览展示包。 |
| `role_roster.yaml` / `enemy_roster.yaml` / `map_roster.yaml` | 本模块允许哪些角色、怪物、地图参与，以及默认推荐与按难度覆盖。 | 角色、怪物、地图、关卡的本体字段与运行时行为正文。 |

### 3.1 `module.yaml` 长期负责

- 模块稳定 ID 与模块版本
- 模块主题、简介、默认地图与默认起点
- 模块正文引用哪些名单载体
- 模块发布状态与兼容边界
- 模块默认难度策略与默认角色策略

### 3.2 长期禁止的混写方式

- 不把所有角色、怪物、地图、关卡名单都塞到 `modules_index.yaml`
- 不把模块入口图标、首页可见性等入口层字段反向塞回 `module.yaml` 作为唯一真源
- 不把模块名单直接写成散乱字符串数组分布在多个正文文件里且无统一名单载体

## 四、模块总配置最小字段契约

建议 `game/data/modules/<module_id>/module.yaml` 至少包含：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 模块稳定 ID |
| `module_version` | 模块版本 |
| `name_zh` | 模块中文名 |
| `summary_zh` | 模块摘要说明 |
| `theme` | 模块主题 |
| `publish_status` | `draft` / `published` / `archived` |
| `visibility_scope` | 正式玩家、调试验证或隐藏范围 |
| `default_map_id` | 默认地图入口 |
| `default_start_node_id` | 默认起点节点 |
| `default_role_id` | 默认推荐角色 |
| `roster_refs` | 名单载体引用集合 |
| `difficulty_profile_ref` | 模块难度策略引用 |
| `unlock_rule_id` | 模块解锁规则 |
| `asset_bundle_ref` | 模块入口展示素材引用 |

### `roster_refs` 最小字段建议

| 字段 | 作用 |
| --- | --- |
| `role_roster_ref` | 角色名单引用 |
| `enemy_roster_ref` | 怪物名单引用 |
| `map_roster_ref` | 地图名单引用 |
| `level_roster_ref` | 关卡集合引用 |

## 五、模块名单载体最小契约

### 5.1 `role_roster.yaml`

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 所属模块 |
| `available_role_ids` | 当前模块可选角色列表 |
| `default_role_id` | 默认推荐角色 |
| `hidden_role_ids` | 本模块隐藏但保留兼容的角色 |
| `difficulty_role_overrides` | 不同难度下的角色可见性覆盖 |
| `preview_role_ids` | 允许在模块局部预览或推荐展示的角色 |

### 5.2 `enemy_roster.yaml`

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 所属模块 |
| `available_enemy_ids` | 当前模块普通怪与精英怪列表 |
| `available_boss_ids` | 当前模块 Boss 列表 |
| `difficulty_enemy_overrides` | 不同难度下的开放覆盖 |
| `default_boss_id` | 默认 Boss 或终点 Boss |

### 5.3 `map_roster.yaml`

建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 所属模块 |
| `available_map_ids` | 当前模块可用地图列表 |
| `default_map_id` | 默认地图 |
| `node_map_bindings` | 模块节点与地图绑定关系 |
| `hidden_map_ids` | 隐藏或验证专用地图 |

## 六、模块可见性与模块状态边界

### 6.1 模块发布状态

长期至少支持：

- `draft`
- `published`
- `archived`

含义：

- `draft`：后台可编辑，默认不进入正式玩家入口。
- `published`：允许进入正式发布包。
- `archived`：历史保留，不再默认展示给正式玩家。

### 6.2 模块可见范围

长期建议至少支持：

- `player_visible`
- `debug_only`
- `hidden`

说明：

- `player_visible`：正式玩家入口可见。
- `debug_only`：仅调试和验证入口可见。
- `hidden`：默认不展示，只用于兼容、迁移或历史保留。

### 6.3 模块状态与名单状态解耦

- 模块本体 `published` 不等于名单中的所有角色、怪物、地图都默认可用。
- 名单载体应允许按难度、按解锁规则或按可见范围对实体做二次过滤。
- 模块归档后，历史名单可保留，但默认不进入正式玩家入口。

## 七、模块正文与名单的协作规则

### 7.1 角色

- 模块正文通过 `role_roster.yaml` 决定本模块允许哪些角色进入。
- 角色固定武器、固定技能和动画包仍由角色正文承载。
- 名单层只决定“本模块允许谁参与”，不决定角色本体属性。

### 7.2 怪物与 Boss

- 模块正文通过 `enemy_roster.yaml` 决定本模块允许哪些普通怪、精英怪和 Boss 参与。
- 怪物技能、Boss 形态、狂暴概率和掉落绑定仍由怪物 / Boss 正文承载。

### 7.3 地图

- 模块正文通过 `map_roster.yaml` 决定本模块允许哪些地图参与节点与关卡绑定。
- 地图分层、区域语义、碰撞区和事件区仍由地图正文承载。

## 八、默认入口与名单的关系

- `module.yaml` 中的 `default_role_id`、`default_map_id` 和 `default_start_node_id` 必须与各名单载体一致。
- 默认推荐角色必须出现在 `available_role_ids` 中。
- 默认地图必须出现在 `available_map_ids` 中。
- 默认起点节点必须能解析到合法关卡节点与地图绑定。

## 九、异常与阻断边界

| 异常场景 | 处理方式 | 最终状态 |
| --- | --- | --- |
| 模块索引存在但 `module.yaml` 缺失 | 视为模块正文缺失 | 阻断发布与进入 |
| 模块正文存在但名单载体缺失 | 视为模块正文不完整 | 阻断发布 |
| 默认推荐角色不在角色名单中 | 视为模块入口契约失效 | 阻断发布 |
| 默认地图不在地图名单中 | 视为模块入口契约失效 | 阻断发布 |
| 名单引用的实体正文不存在 | 视为引用断链 | 阻断发布 |
| 模块归档后仍默认暴露给正式玩家 | 视为可见性错误 | 回退修正 |

## 十、发布前最小校验集合

| 校验项 | 通过条件 | 阻断条件 |
| --- | --- | --- |
| 模块总配置完整性 | `module_id`、主题、状态、默认入口字段齐全 | 关键字段缺失 |
| 名单载体完整性 | 角色、怪物、地图名单引用可解析 | 名单缺失或断链 |
| 默认入口合法性 | 默认角色、地图、节点都能落到合法对象 | 默认入口失效 |
| 可见范围合法性 | 正式玩家、调试验证和隐藏范围清晰 | 模块暴露范围混乱 |
| 名单与正文一致性 | 名单项都能映射到合法角色 / 怪物 / 地图正文 | 实体引用断链 |
| 发布状态与入口一致性 | `draft` / `archived` 不误入正式入口 | 状态越界 |

## 十一、与现有文档的关系

- [附录-模块索引与启动入口规范](附录-模块索引与启动入口规范.md) 负责入口层边界；本文只承接入口层之后的模块本体与名单载体。

## 十二、结论

- 本文只保留三层契约：`modules_index.yaml` 之后的 `module.yaml`、名单载体和默认入口一致性校验。
- 角色、怪物、地图和关卡的本体字段继续由各自专题维护，不在本文扩写第二遍。

## 十三、本文不负责的事项

- 不负责首页主入口、首页预览展示包和公共导航字段。
- 不负责继续上次、开始新游戏和难度入口的入口层决策。
- 不负责角色固定武器、技能行为、投射物、状态效果和关卡波次正文。
