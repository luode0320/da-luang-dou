# 发布记录回滚记录与存档元数据契约

## 目标

本附录用于把以下三类字段级载体收口为长期架构契约：

- 发布记录
- 回滚记录
- 存档元数据

上一轮已经在 [附录-版本矩阵与兼容判定规范](附录-版本矩阵与兼容判定规范.md) 中统一了版本口径和兼容判定规则；本附录继续下钻，把“这些字段最终落在哪、最少有哪些、谁来写、谁来读”写成最小契约，避免后续后台发布器、回滚器、游戏读取器和测试验证继续各自补表。

## 适用范围

- `admin/releases/<release_id>/`
- `game/data/published/<release_version>/`
- `user://saves/`
- 后续后台发布与回滚服务
- 后续游戏存档恢复与兼容判定逻辑

## 一、三类载体的职责边界

| 载体 | 核心职责 | 主要写入方 | 主要读取方 | 不负责 |
| --- | --- | --- | --- | --- |
| 发布记录 | 记录一次发布到底发布了什么、依赖哪些版本、校验是否通过 | 后台发布器 | 后台管理界面、测试验证、回滚器 | 不直接替代运行时 `manifest.yaml` |
| 回滚记录 | 记录一次回滚从哪里回到哪里、为什么回滚、验证是否通过 | 后台回滚器 | 后台管理界面、审计、测试验证 | 不替代发布记录本体 |
| 存档元数据 | 记录某个模块自动存档与哪组版本绑定、能否继续恢复 | 游戏运行层 | 游戏读取器、兼容判定器、调试验证 | 不保存完整业务正文快照 |

统一约束：

1. 发布记录是版本矩阵的业务审计入口。
2. 回滚记录是版本切换历史的审计入口。
3. 存档元数据是存档恢复判定入口。
4. 三者都必须引用稳定版本字段，不使用展示名或页面临时文案做恢复依据。

## 二、发布记录最小契约

### 2.1 目录建议

长期建议发布记录目录结构如下：

```text
admin/releases/<release_id>/
├── summary.yaml
├── checks.yaml
├── matrix.yaml
└── audit.yaml
```

说明：

- `summary.yaml` 记录一次发布的主摘要
- `checks.yaml` 记录发布前校验结果
- `matrix.yaml` 记录本次发布对应的版本矩阵
- `audit.yaml` 记录操作与审计信息

### 2.2 summary.yaml 最小字段

| 字段 | 作用 |
| --- | --- |
| `release_id` | 本次发布记录稳定 ID |
| `config_version` | 本次内容发布版本 |
| `release_version` | 本次发布包版本，默认与 `config_version` 一一对应 |
| `game_version` | 本次目标游戏构建版本 |
| `config_schema_version` | 本次发布包结构版本 |
| `status` | 例如 `published`、`failed`、`rollback_target` |
| `created_at` | 发布时间 |
| `created_by` | 发布执行人 |
| `release_notes` | 本次发布摘要 |
| `previous_release_version` | 上一个稳定发布版本 |
| `rollback_target_id` | 默认可回滚目标 |

### 2.3 checks.yaml 最小字段

| 字段 | 作用 |
| --- | --- |
| `release_id` | 对应发布记录 ID |
| `validation_status` | 总体校验结果 |
| `blocking_issues` | 阻断项 |
| `warning_issues` | 警告项 |
| `smoke_targets` | 建议冒烟验证目标 |
| `save_compatibility_status` | 存档兼容结论 |
| `generated_at` | 校验生成时间 |

### 2.4 matrix.yaml 最小字段

| 字段 | 作用 |
| --- | --- |
| `release_id` | 对应发布记录 ID |
| `game_version` | 当前游戏构建版本 |
| `config_version` | 当前配置版本 |
| `release_version` | 当前发布包版本 |
| `module_versions` | 本次发布包含的模块版本列表 |
| `asset_versions` | 本次发布包含的素材版本列表 |
| `save_schema_version` | 当前存档结构版本 |
| `required_game_version` | 当前发布包要求的最小游戏版本 |

### 2.5 audit.yaml 最小字段

| 字段 | 作用 |
| --- | --- |
| `release_id` | 对应发布记录 ID |
| `operator_id` | 操作者稳定 ID |
| `operator_name` | 操作者展示名 |
| `operation_type` | 例如 `publish` |
| `started_at` | 开始时间 |
| `finished_at` | 完成时间 |
| `result` | 成功或失败 |
| `change_summary` | 变更摘要 |

## 三、回滚记录最小契约

### 3.1 目录建议

长期建议回滚记录沿用发布记录根目录，在对应 `release_id` 下额外写入：

```text
admin/releases/<release_id>/
└── rollback.yaml
```

如果一次回滚不是从当前发布记录直接发起，也应生成新的 `release_id` 作为回滚操作记录，而不是只改旧记录状态。

### 3.2 rollback.yaml 最小字段

| 字段 | 作用 |
| --- | --- |
| `rollback_id` | 本次回滚记录稳定 ID |
| `trigger_release_id` | 发起回滚时所在的发布记录 ID |
| `from_release_version` | 当前版本 |
| `to_release_version` | 目标回滚版本 |
| `to_config_version` | 目标配置版本 |
| `reason` | 回滚原因 |
| `affected_module_ids` | 受影响模块 |
| `save_compatibility_status` | 与当前活跃存档的兼容结论 |
| `validation_status` | 回滚前判定结果 |
| `rollback_result` | 回滚执行结果 |
| `verified_by` | 回滚验证人 |
| `verified_at` | 回滚验证时间 |
| `verification_notes` | 回滚验证摘要 |
| `created_at` | 回滚发起时间 |
| `created_by` | 回滚执行人 |

### 3.3 回滚记录的最小验证证据

回滚记录不只记录“切了版本”，还必须能沉淀：

- 回滚前判定结果
- 回滚后验证摘要
- 验证人和验证时间
- 是否存在需要人工修复的兼容问题

至于“回滚验证到底最少覆盖哪些路径”，统一转交 [附录-发布验证与放行闭环规范](附录-发布验证与放行闭环规范.md)；本文只负责这些结论最终如何挂到记录载体上。

## 四、存档元数据最小契约

### 4.1 设计原则

存档元数据只负责回答“这个存档绑定了哪组版本、能不能继续恢复”，不负责复制整份发布包正文。

### 4.2 模块自动存档元数据字段

| 字段 | 作用 |
| --- | --- |
| `save_schema_version` | 当前存档结构版本 |
| `module_id` | 当前模块稳定 ID |
| `module_version` | 当前模块写入时对应的模块版本 |
| `last_game_version` | 最后一次写入该存档时的游戏版本 |
| `last_config_version` | 最后一次写入该存档时的配置版本 |
| `last_release_version` | 最后一次写入该存档时的发布包版本 |
| `last_required_game_version` | 最后一次写入时发布包要求的最小游戏版本 |
| `last_saved_at` | 最近一次保存时间 |
| `current_node_id` | 当前节点 |
| `current_level_id` | 当前关卡 |
| `difficulty_id` | 当前难度 |
| `role_id` | 当前角色 |

### 4.3 推荐字段

| 字段 | 作用 |
| --- | --- |
| `equipped_external_item_ids` | 当前外部永久金色道具搭配 |
| `module_temp_item_ids` | 当前模块内临时道具 |
| `module_temp_currency` | 当前模块内临时货币 |
| `last_resume_status` | 最近一次恢复是否成功 |
| `last_resume_warning` | 最近一次恢复警告 |

### 4.4 长期资产元数据字段

长期资产记录至少建议保留：

| 字段 | 作用 |
| --- | --- |
| `profile_progress_version` | 长期进度结构版本 |
| `owned_permanent_item_ids` | 已获得永久金色道具 |
| `unlocked_module_ids` | 已解锁模块 |
| `cleared_level_ids` | 已通关关卡 |
| `unlocked_role_ids` | 已解锁角色 |

## 五、字段之间的引用关系

统一引用规则如下：

1. 发布记录通过 `release_id` 自身做后台审计主键。
2. 发布记录通过 `release_version` / `config_version` 与发布包目录和版本矩阵对应。
3. 回滚记录通过 `from_release_version` / `to_release_version` 与发布记录、发布包和版本矩阵对应。
4. 存档元数据通过 `last_release_version`、`last_config_version`、`module_version` 与版本矩阵对应。
5. 存档元数据中的 `module_id`、`current_node_id`、`current_level_id`、`role_id` 都必须是稳定 ID。

## 七、阻断条件

以下情况默认阻断：

- 发布记录缺少 `release_id`、`config_version`、`release_version` 或 `game_version`
- 回滚记录缺少 `from_release_version`、`to_release_version` 或 `rollback_result`
- 存档元数据缺少 `save_schema_version`、`module_id`、`last_release_version`
- 存档元数据缺少恢复判定所需的关键版本引用
- 回滚记录缺少兼容结论字段却宣称验证通过

## 九、结论

- 本文只维护发布记录、回滚记录和存档元数据的字段级落盘契约，以及这些载体的最小读取职责边界。
- 三态边界、版本兼容口径和最终放行门槛继续转交相邻发布专题，不在本文重复定义验证覆盖范围、兼容判定规则或发布/回滚/恢复验证步骤顺序。

## 本文不负责的事项

- 不负责版本对象的统一命名与兼容判定规则本体。
- 不负责 `manifest.yaml`、`compatibility.yaml` 和索引文件的最小字段正文。
- 不负责最终放行结论分级与验证证据覆盖要求。
