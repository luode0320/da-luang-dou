# 业务链路索引页再次压回阅读入口

## 背景

- `4-主要业务链路.md` 之前已经收口为索引页，但继续复核后，活动链路表仍保留了“用途摘要 / 入口 / 责任模块”这类单链路文档开头信息。
- 这样虽然不算正文级重复，但索引页仍然在维护一份缩略版的“单链路头部元信息”，后续继续演进时仍可能出现双处同步成本。

## 本轮收口目标

- 让 `4-主要业务链路.md` 更接近纯阅读入口页。
- 把业务目标、责任模块和完整入口清单继续压回 `5-9` 单链路文档。
- 保留索引页最小必要的信息：当前链路主要回答什么、正式入口在哪里、当前是否启用。

## 本轮实际调整

### `4-主要业务链路.md`

- 从“本文负责什么”中删除“说明每条链路的业务目标摘要、入口和责任模块”。
- 在“本文不负责什么”中补充：索引页不再重复维护业务目标、责任模块和完整入口清单。
- 将“活动链路”表从：
  - `用途摘要 / 入口 / 责任模块 / 状态`
  收紧为：
  - `当前主要回答的问题 / 正式入口 / 状态`
- 在“索引页与单链路分工提醒”中明确：业务目标、责任模块、先后顺序、入口切换、输入输出、异常分支和变更影响统一回到单链路文档。

## 本轮边界结论

| 文档 | 本轮收紧后的唯一正文职责 | 不再维护 |
| --- | --- | --- |
| `4-主要业务链路.md` | 当前有哪些链路、各自主要回答什么、正式入口在哪、应该先读哪条 | 业务目标摘要、责任模块清单、完整入口清单 |
| `5-9` 单链路文档 | 各自链路的业务目标、责任模块、流程顺序、异常、输入输出和变更影响 | 索引排序与阅读入口规则 |

## 结论

- 这轮之后，`4-主要业务链路.md` 更接近纯“阅读入口与状态索引页”。
- 后续如果单链路继续扩展，优先更新单链路文档本身；只有当“该读哪条链路、当前是否启用、编号是否变化”发生变化时，才回到索引页更新。
