# 后台发布器与游戏读取器目标分层

## 目标

本附录把“后端怎么产出发布包、游戏怎么消费发布包”进一步收口到目标代码分层，方便后续真正开始写 Go 和 GDScript 时不再临时拼装。

这里描述的是目标分层设计，不代表当前仓库已经存在这些代码目录。

## 分层边界速查

| 主题 | 本文负责 | 应转交的专题 |
| --- | --- | --- |
| 目标代码层级 | Go 发布器和 Godot 读取器后续应拆哪些层、哪些组件。 | 三态职责划分转交 [附录-发布包契约与三态边界](附录-发布包契约与三态边界.md)。 |
| 交接面 | 发布器和读取器之间长期只能通过哪些正式契约入口与正文载体交接。 | 发布版本入口、兼容契约入口、索引入口的最小字段正文转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。 |
| 实现依赖关系 | 发布器、读取器应分别依赖哪些版本、记录和验证规则。 | 版本兼容、记录载体和放行门槛分别转交版本矩阵、记录契约和放行闭环专题。 |

## 一、后台发布器目标分层

后台发布器的职责不是简单复制 YAML 文件，而是把编辑态草稿整理成可校验、可追踪、可回滚的发布契约态。

### 建议分层

| 层级 | 建议目录 | 职责 |
| --- | --- | --- |
| 入口层 | `main.go` | 启动服务、装配依赖、注册路由 |
| 路由层 | `internal/router/` | HTTP 路由注册与中间件装配 |
| 控制器层 | `internal/controller/` | 接收草稿、校验、发布、回滚等请求 |
| 服务层 | `internal/service/` | 承载发布器核心业务流程 |
| 契约实体层 | `internal/entity/` | 请求体、响应体、发布包契约结构 |
| 存储层 | `common/repository/` 或 `internal/repository/` | 草稿、发布记录、回滚版本、审计记录读写 |

### 服务层建议继续拆分

| 服务 | 作用 |
| --- | --- |
| `draft_service` | 管理草稿保存、草稿装载、草稿比较 |
| `validation_service` | 执行字段、引用、素材、节点、兼容校验 |
| `publish_service` | 组装发布契约关键载体与正文配置并生成发布包 |
| `rollback_service` | 回滚到历史稳定版本 |
| `audit_service` | 记录发布、回滚、校验操作审计 |
| `preview_service` | 为后台预览中心生成预览态结果 |

### 契约实体建议

至少应有以下结构类别：

- 发布版本入口结构
- 兼容契约入口结构
- 模块索引项结构
- 实体索引项结构
- 素材索引项结构
- 发布请求结构
- 发布结果结构
- 校验结果结构

## 二、游戏读取器目标分层

游戏读取器的职责不是“到处随手读 YAML”，而是基于发布契约构建稳定加载链。

### 建议分层

| 层级 | 建议目录 | 职责 |
| --- | --- | --- |
| 启动层 | `game/bootstrap/` | 游戏启动、版本选择、配置系统初始化 |
| 契约读取层 | `game/runtime/config/io/` | 读取发布版本入口、兼容契约入口和索引入口 |
| 合并层 | `game/runtime/config/merge/` | 处理全局、模块、实体覆盖与默认值 |
| 缓存层 | `game/runtime/config/cache/` | 模块缓存、关卡缓存、实体缓存 |
| 兼容层 | `game/runtime/config/compat/` | 字段别名、废弃字段、ID 重定向处理 |
| 追踪层 | `game/runtime/config/trace/` | 记录发布版本、覆盖来源、警告与错误 |

### 读取器核心组件建议

| 组件 | 作用 |
| --- | --- |
| `release_selector` | 确定当前读取哪个发布版本 |
| `release_entry_loader` | 读取并校验当前发布版本入口 |
| `compatibility_entry_loader` | 读取兼容契约入口 |
| `index_entry_loader` | 读取模块、实体、素材索引入口 |
| `config_locator` | 根据索引定位正文配置 |
| `merge_service` | 应用覆盖与默认值 |
| `module_cache` | 管理模块级缓存 |
| `entity_cache` | 管理实体级缓存 |
| `config_trace` | 输出当前版本、路径和异常追踪 |

## 三、后台与游戏之间的明确交接面

后台发布器和游戏读取器之间，长期只通过以下交接面通信：

- 发布版本入口
- 兼容契约入口
- 索引入口
- 已发布正文配置

不允许通过以下内容建立隐式耦合：

- 后台页面结构
- 后台数据库表单布局
- 临时预览数据
- 草稿目录层级

## 四、阶段 0 的最小实现闭环

如果只做最小闭环，后台和游戏至少分别完成下面这些部分：

### 后台后端最小闭环

1. 装载草稿
2. 执行基础校验
3. 生成最小发布契约关键文件
4. 生成最小索引集合
5. 写入标准化发布目录
6. 输出最小发布结果摘要

### 游戏侧最小闭环

1. 识别目标发布版本
2. 读取发布契约关键文件
3. 读取最小索引集合
4. 按模块定位正文配置
5. 生成运行时缓存
6. 输出最小追踪信息

## 五、对当前 README 骨架的直接约束

- 合并层
