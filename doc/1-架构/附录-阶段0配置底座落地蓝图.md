## 一、阶段 0 的长期定位

- 阶段 0 不是玩家可见功能，而是整个项目的配置底座。
- 它负责把“后台生产内容 -> 版本化发布包 -> 游戏运行时只读发布结果”的主链路先落成最小闭环。
- 只要阶段 0 没有闭环，阶段 1-7 的首页、模块、地图、角色、怪物、关卡、发布和验证都会缺统一真相源。

## 二、阶段 0 完成时必须成立的最小闭环

阶段 0 的长期完成标准，不看页面多少，也不看素材多少，而看以下闭环是否已经成立：

1. 后台能装载草稿并执行基础校验。
2. 后台能把草稿转换为版本化发布包。
3. 发布包至少具备发布版本入口、兼容契约入口和最小索引入口集合。
4. 游戏运行时只读取发布包，不直接读取后台草稿。
5. 游戏运行时能按统一顺序读取、合并、缓存并输出追踪信息。
6. 发布、回滚、兼容与恢复都指向同一套版本口径。

## 三、阶段 0 的最小架构蓝图

### 3.1 后台发布器最小蓝图

后台后端最小只需要先打通下面这条链：

1. 草稿装载
2. 字段与引用校验
3. 发布契约组装
4. 标准化目录写出
5. 发布记录落盘

后续实现时，推荐最少按以下职责拆开：

| 层级 | 最小职责 |
| --- | --- |
| 入口层 | 启动服务、装配依赖、暴露发布与校验入口。 |
| 草稿层 | 管理草稿读取、差异比较和来源信息。 |
| 校验层 | 校验字段、引用、素材、节点、难度和兼容风险。 |
| 发布层 | 生成发布版本入口、兼容契约入口、索引入口和标准化正文目录。 |
| 记录层 | 写入发布记录、回滚记录和审计信息。 |

### 3.2 游戏读取器最小蓝图

游戏侧最小只需要先打通下面这条链：

1. 选择目标发布版本
2. 读取发布版本入口
3. 读取兼容契约入口
4. 读取最小索引入口集合
5. 定位正文配置
6. 执行覆盖合并
7. 建立模块级和关卡级缓存
8. 输出追踪与错误定位信息

后续实现时，推荐最少按以下职责拆开：

| 层级 | 最小职责 |
| --- | --- |
| 启动层 | 选择发布版本并初始化配置系统。 |
| 契约读取层 | 读取发布版本入口、兼容契约入口和索引入口。 |
| 合并层 | 处理全局默认、模块覆盖、实体覆盖与内置默认值。 |
| 缓存层 | 管理索引缓存、模块缓存、关卡缓存与实体缓存。 |
| 追踪层 | 输出版本、路径、覆盖来源、缺失项和警告项。 |

## 四、阶段 0 的发布包最小实现闭环

长期上，阶段 0 最少要把以下闭环打通：

1. 版本化发布目录可被稳定定位。
2. 发布版本入口可提供当前版本、关键路径和关键版本字段。
3. 兼容契约入口可提供当前读取与恢复所需的最小兼容信息。
4. 索引入口集合可把模块、实体、素材正文定位到正式路径。
5. 正文配置目录可承载 `control/`、`global/`、`modules/` 三类已发布内容。
6. 游戏运行时可按“入口 -> 索引 -> 正文 -> 合并 -> 缓存 -> 追踪”的统一顺序完成读取。

这些入口的具体目录树、文件名、字段清单和精确读取顺序，统一转交 [附录-发布包最小字段契约](附录-发布包最小字段契约.md)。

## 六、阶段 0 完成前后，文档分工应保持什么边界

- `doc/1-架构/` 负责长期规则、最小契约、目标分层和依赖关系。
- `doc/3-实施/` 负责实施步骤、阶段任务、验证点、风险与阻断。
- `doc/5-tests/` 负责后续真实验证资产和回归证据。
- `doc/7-验收/` 负责前置验收标准和最终放行结论。

换句话说，阶段 0 的“为什么这样设计、最小闭环长什么样”放在这里；阶段 0 的“今天怎么做、做到了哪一步”继续留在实施文档。

## 八、结论

- 阶段 0 的长期真相源不应只停留在实施文档。
- 后续开始写 Go 后端骨架、Vue 后台页面或 Godot 配置读取器前，应先以本文确认最小闭环和依赖边界。
