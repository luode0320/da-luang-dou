# 文档索引入口与截图归类导航再收口

## 背景

- 这轮不是继续改业务正文，而是继续收紧文档导航层。
- 截图里的文件归类本身已经稳定，但 `doc/README.md`、`doc/1-架构/README.md`、`doc/6-审查/README.md` 仍然容易把“正式入口”和“整理过程归档”平铺在一起。
- 这会让后续继续查旧截图文档时，需要在多份近义审查说明之间来回切换。

## 本轮收口目标

- 总导航只回答“正式目录是什么、当前主入口是什么、截图旧文档现在应该跳到哪里”。
- 架构目录导航只回答“先看哪一组、旧架构专题现在由哪份正式附录承接”。
- 主链路索引只回答“这条链路主要解决什么问题、正式入口在哪、是否启用”。
- 审查目录导航只保留当前优先入口，历史同日收口文档继续保留为过程归档，不再平铺成第二份总导航。

## 本轮调整

### `doc/README.md`

- 保留正式目录职责、当前主入口和截图旧文档速查。
- 审查区只保留三类当前入口：
  - 截图文档逐项归类对照表
  - 截图文档目录规则复核与最终归档
  - 本文
- 不再把同日多份目录整理说明同时挂到总导航。

### `doc/1-架构/README.md`

- 继续保留：
  - 先看哪一组
  - 稳定主入口
  - 业务链路入口
  - 高重叠专题速查
  - 历史专题承接映射
- 进一步收紧高重叠专题速查里的表述，只保留“哪份附录回答什么边界、不负责什么”。
- 明确截图旧 `2026-06-22_*` 架构专题只通过“历史专题承接映射”跳到正式附录，不在目录页并行保留 dated 入口。

### `doc/1-架构/4-主要业务链路.md`

- 活动链路表继续只保留：
  - 当前主要回答的问题
  - 正式入口
  - 状态
- 将问题描述压回单句阅读提示，不再偷带单链路正文细节。
- 修正活动链路表的列数，使索引页格式与职责一致。

### `doc/6-审查/README.md`

- 从“平铺大量同日整理文档”收口为“当前优先入口 + 历史归档说明”。
- 后续如果用户继续问截图归类或目录规则，默认先从 README 指向的少量主入口进入，不再先翻整目录。

## 当前单一真相源

- 看正式目录职责：`doc/README.md`
- 看截图文件逐项归类：`doc/6-审查/2026-06-28_截图文档逐项归类对照表.md`
- 看这批截图文件为什么仍然是 7 类正式目录：`doc/6-审查/2026-06-28_截图文档目录规则复核与最终归档.md`
- 看旧架构 dated 专题现在对应哪份正式附录：`doc/1-架构/README.md`
- 看主链路应该先读哪条：`doc/1-架构/4-主要业务链路.md`

## 结论

- 这轮没有再迁移业务文档归属，重点是把导航层继续压回“入口选择”和“归档分层”。
- 当前默认使用方式已经收紧为：
  - 正式目录入口承接业务正文
  - 审查目录承接整理过程与规则复核
  - README 只保留当前优先入口，不继续堆平行说明
