# 模块职责

## 文档说明

- 用途: 明确当前项目各长期模块的职责、输入输出、依赖方向和协作边界，避免内容生产、配置发布和游戏运行职责混写。
- 事实来源: `项目设计.md`、`README.md`、`AGENTS.md`、`game/data/README.md`、`admin/README.md`
- 适用范围: `admin/`、`game/`、`game/data/`、`tools/windows/`、`doc/`
- 最近同步时间: 2026-06-28 20:05:00

## 本文负责什么

- 说明长期模块各自负责什么、不负责什么。
- 说明模块的主要输入、主要输出和依赖方向。
- 说明后台编辑、发布契约、运行时、进度恢复和 Windows 支持之间如何协作。
- 作为模块级真相源，回答“谁对什么结果负责”。

## 本文不负责什么

- 不展开目录树和路径允许内容，那属于 [目录树](2-目录树.md)。
- 不展开 `module.yaml`、`role_roster.yaml`、`home.yaml`、`manifest.yaml` 等具体载体字段，那属于对应附录。
- 不展开链路顺序、异常分支和入口切换，那属于 [主要业务链路](4-主要业务链路.md) 与各业务链路文档。

## 依赖方向

- 项目总览 -> 架构专题 -> 需求 / 实施 / 审查文档引用，不反向由业务文档决定长期模块边界。
- 后台前端 -> 后台后端 -> 发布契约层 -> 游戏运行层，为配置内容进入运行时的主方向。
- 控制配置层、全局配置层、模块配置层先经后台后端校验与发布，再进入发布契约层，不直接越过发布流程进入运行时。
- 进度与存档层依赖游戏运行结果和发布版本事实，不反向定义发布契约。
- Windows 发布支持依赖游戏构建产物和发布产物，不反向决定玩法和配置规则。

## 协作边界

- 后台编辑态、发布契约态、游戏运行态必须保持单向流转，不允许运行态反向写回发布包正文。
- 后台配置系统只负责“编辑、校验、预览、发布、回滚”，不负责游戏运行时表现。
- 后台后端内部继续拆层时，优先把“校验”“发布”“回滚”“审计”拆成独立服务，而不是堆进一个超大 service。
- 游戏运行层只负责“读、合、用”，不负责编辑、目录管理和素材上传。
- 游戏读取器继续落代码时，优先把“版本选择”“契约读取”“合并”“缓存”“兼容”“追踪”拆成独立组件，而不是在单个 loader 中混写。
- 版本矩阵应由发布契约层统一出具，后台界面、游戏读取器和测试验证都引用这同一套版本事实，不各自派生第二套版本口径。
- 发布验证与最终放行门槛应由发布契约层、进度与存档层、游戏运行层和 Windows 发布支持共同参与，不得只在某一个后台页面里口头给出“可发布”。
- 首页设置默认值与玩家个人设置必须分层；后台负责默认值，运行时负责个人偏好，不得混成单一真源。
- 控制配置层只负责入口层与控制层，不负责角色、怪物、技能、武器、道具等实体正文。
- 全局配置层只负责跨模块默认值与公共实体，不负责单模块覆盖和单模块专属内容。
- 模块配置层只负责单模块内容组织和单模块覆盖，不负责跨模块公共默认值。
- 控制层、公共层、模块层的具体路径、载体命名和字段结构，统一转交 [目录树](2-目录树.md) 与相邻附录，不在本表重写第二遍。
- 角色绑定、技能本体、固定武器、投射物、状态效果、道具掉落和地图关卡仍需继续分专题维护，不在本表重新展开载体和字段细节。
- 后台发布状态与玩家进度状态必须分层，不能用同一字段同时表达“是否已发布”和“玩家是否已通关”。
- 进度与存档层只负责恢复入口、长期资产与模块内临时资产生命周期，不直接承担配置发布职责。
- `开始新游戏` 只重置当前模块的模块级进度与临时资产，不清空长期资产。
- 配置读取合并顺序、覆盖优先级和运行时缓存策略，统一转交 [附录-游戏侧配置读取与合并规范](附录-游戏侧配置读取与合并规范.md)，不在本表平行维护另一套规则。
- 文档层中，需求、实施、Bug、测试、审查、验收分别归各自目录，长期架构规则只留在 `doc/1-架构/`。
- 当截图旧文档或阶段文档里同时含有“长期规则 + 当前计划”两种内容时，长期规则应抽回 `doc/1-架构/`，当前计划继续留在 `doc/3-实施/` 或 `doc/2-需求/`。

## 变更同步

- 模块新增、拆分、合并或职责迁移时，优先更新本文件。
- 目录职责变化时，同步更新 [目录树](2-目录树.md)。
- 发布契约、版本兼容、发布器/读取器分层、放行与回滚职责变化时，同步更新对应发布专题附录。
- 首页入口、模块入口、模块正文、读取合并和恢复入口边界变化时，同步更新控制配置层与生命周期相关附录。
- 角色、怪物、技能、武器、状态效果、道具、地图和关卡等实体专题边界变化时，同步更新对应专题附录。
- 若模块协作边界变化影响顶层理解，同步更新 [总架构](1-总架构.md) 与 `项目设计.md` 摘要。
- 若截图旧专题、历史需求文档或阶段文档中的长期规则被正式抽回本目录体系，同步更新 `doc/README.md` 与相关审查归档说明，避免导航仍指向旧入口。
