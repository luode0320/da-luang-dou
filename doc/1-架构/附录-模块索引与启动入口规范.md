# 模块索引与启动入口规范

## 目标

本附录用于把阶段 2 中已经提出、但尚未在长期架构层写硬的“模块索引、模块状态、解锁规则、继续上次、开始新游戏、难度入口和模块入口展示”统一成长期规则。

它重点回答以下问题：

- 模块索引应该承载哪些最小字段
- 后台发布状态与玩家可见进度状态如何分层
- `继续上次` 与 `开始新游戏` 在模块入口层到底如何判定为合法入口
- 模块入口、难度入口、解锁规则和归档隐藏如何进入统一配置契约
- 模块选择页、模块地图和关卡节点页在入口层各自扮演什么角色

- 模块选择页、模块地图页、关卡节点页
- 每模块单一自动存档入口

## 一、分层总原则

### 1. 模块索引是“入口层配置”，不是模块正文替代品

- 模块索引负责“有哪些模块、模块当前能不能进、从哪里进、显示什么”。
- 模块正文负责“这个模块内部有哪些地图、节点、角色、怪物、关卡和规则”。
- 不允许把整套模块正文字段平铺复制进模块索引。

### 2. 后台发布状态与玩家进度状态必须分层

- 后台发布状态用于表达某个模块配置是否处于草稿、已发布或归档状态。
- 玩家进度状态用于表达某个玩家视角下该模块是否未解锁、已解锁、进行中、已通关。
- 这两个状态可以有关联，但不能混成同一字段。

### 3. 模块入口层只回答“是否可进、有哪些合法入口、默认进入模式是什么”

模块入口层长期只回答以下问题：

1. 首页是否展示该模块
2. 玩家是否满足进入条件
3. 当前默认进入模式属于模块地图、节点页还是恢复判断链路
4. 是否存在当前模块自动存档恢复入口
5. 哪些难度当前可见、可选或被锁定

入口层不负责：

- 恢复后具体先进入哪个流程节点
- 开始新游戏后的实际跳转顺序
- 进入关卡前的统一确认步骤

这些顺序性问题继续转交 [9-业务链路-模块启动恢复与验证入口](9-业务链路-模块启动恢复与验证入口.md) 与 [6-业务链路-模块闯关与进度推进](6-业务链路-模块闯关与进度推进.md)。

## 二、入口层对象

长期建议至少区分以下对象：

| 对象 | 职责 | 典型载体 |
| --- | --- | --- |
| 模块索引项 | 表达模块在选择页的入口信息 | `modules_index.yaml` |
| 玩家模块状态 | 表达某玩家对该模块的进度和解锁情况 | 自动存档与长期进度记录 |
| 难度入口配置 | 表达普通、噩梦、地狱三档入口是否开放 | 模块总配置或模块索引项 |

说明：

- `modules/<module_id>/module.yaml` 属于模块正文与名单层长期真相源，由 [附录-模块正文配置与模块名单规范](附录-模块正文配置与模块名单规范.md) 负责。
- 本文只在入口决策需要时引用 `module.yaml` 的结果，不在本文重复定义它的字段正文。

## 三、状态分层

### 3.1 后台发布状态

后台发布状态最少建议包括：

- `draft`
- `published`
- `archived`

含义：

- `draft`：后台可编辑，默认不进入正式玩家入口。
- `published`：可被正式发布包引用。
- `archived`：历史保留，不再进入默认模块选择列表。

### 3.2 玩家模块进度状态

玩家侧最少建议包括：

- `locked`
- `unlocked`
- `in_progress`
- `cleared`

含义：

- `locked`：当前玩家不满足进入条件。
- `unlocked`：已可进入，但还没有形成有效进度。
- `in_progress`：存在当前模块的单一自动存档或未完成模块进度。
- `cleared`：已完成模块核心通关目标。

### 3.3 状态关系约束

- 后台 `archived` 的模块默认不应在正式玩家入口展示。
- 后台 `published` 不等于玩家一定 `unlocked`。
- 玩家 `in_progress` 必须以当前模块单一自动存档为依据，不允许伪造。
- 玩家 `cleared` 不等于模块不可重玩；它只表示长期进度已达成。

## 四、模块索引最小字段

建议 `modules_index.yaml` 的每个模块索引项至少包含：

| 字段 | 作用 |
| --- | --- |
| `module_id` | 模块稳定 ID |
| `display_name` | 模块显示名 |
| `description` | 模块简介 |
| `theme` | 模块主题 |
| `publish_status` | 后台发布状态 |
| `default_entry_mode` | 默认进入模式，如模块地图或节点页 |
| `cover_asset_id` | 模块封面素材 |
| `icon_asset_id` | 模块入口图标 |
| `unlock_rule_id` | 解锁规则引用 |
| `difficulty_ids` | 当前模块支持的难度入口列表 |
| `display_order` | 模块展示顺序 |
| `visible_in_home` | 是否在首页或模块选择页展示 |

说明：

- 玩家进度状态不建议直接写进发布包固定索引项，而应由运行时结合存档和长期进度计算。
- `default_entry_mode` 用于决定默认进入模块地图、节点页或恢复入口判断流程。

## 五、模块正文依赖关系

- `modules_index.yaml` 只负责入口显示、解锁与启动决策，不在本文重复定义 `module.yaml` 和名单载体的字段正文。
- `modules/<module_id>/module.yaml`、`role_roster.yaml`、`enemy_roster.yaml`、`map_roster.yaml` 的最小字段契约，统一转交 [附录-模块正文配置与模块名单规范](附录-模块正文配置与模块名单规范.md)。
- 本文只要求入口层在决策 `继续上次`、`开始新游戏`、默认难度和默认进入模式时，能够引用到合法的模块正文结果。

## 六、继续上次 / 开始新游戏入口规则

### 6.1 继续上次

`继续上次` 入口层语义：

- 只有当前模块存在合法自动存档时，才展示为可继续入口。
- 恢复目标不是任意历史节点，而是当前模块唯一自动存档对应的合法继续入口。
- 若自动存档损坏、版本不兼容或引用失效，应回退到阻断或降级提示，不得静默跳到错误节点。
- 本附录只定义“继续上次入口是否成立”；恢复后的步骤顺序继续转交启动恢复主链路与生命周期专题。

### 6.2 开始新游戏

`开始新游戏` 入口层语义：

- 目标是回到当前模块的默认起点节点。
- 只重置当前模块进度与模块内临时资产。
- 不清空长期资产、已解锁模块或永久金色道具。
- 本附录只定义“新游戏入口是否合法、默认起点如何确定”；真正的重置后跳转和后续进入顺序继续转交相邻单链路正文。

### 6.3 默认入口判定口径

长期建议入口层至少能稳定给出以下判定结果：

1. 模块是否对当前玩家可见且已解锁
2. 当前是否存在合法 `继续上次`
3. `开始新游戏` 是否允许进入默认起点
4. 当前默认进入模式属于模块地图、节点页还是恢复判断链路

真正的“先判断什么、后跳去哪里、进入关卡前统一确认什么”继续由 [9-业务链路-模块启动恢复与验证入口](9-业务链路-模块启动恢复与验证入口.md) 负责。

## 七、难度入口规范

- 难度入口属于模块入口层的一部分，不属于关卡正文配置。
- 当前模块最少支持：
  - `normal`
  - `nightmare`
  - `hell`
- 模块入口层只决定“是否开放、是否可见、默认选中哪一档”。
- 具体数值、怪物形态开放、Boss 狂暴概率和奖励差异，由关卡或规则配置承接。

## 八、解锁规则规范

模块入口层至少支持以下解锁语义：

- 默认解锁
- 依赖其他模块通关
- 依赖关卡或节点完成
- 调试直接解锁

统一约束：

- 解锁规则只决定是否允许进入，不直接定义战斗数值。
- 调试直接解锁只服务开发、测试和验证，不作为正式玩家默认路径。

## 九、异常与阻断边界

| 异常场景 | 处理方式 | 最终状态 |
| --- | --- | --- |
| 模块索引存在，但模块正文缺失 | 视为配置不完整 | 阻断发布或阻断进入 |
| 模块已发布，但解锁规则缺失 | 视为入口契约不完整 | 阻断发布 |
| 玩家存在自动存档，但入口恢复目标不存在 | 视为恢复契约失效 | 阻断继续入口，提示修复 |
| `开始新游戏` 清空长期资产 | 视为实现错误 | 阻断上线 |
| 模块归档后仍默认展示给正式玩家 | 视为入口层错误 | 回退修正 |

## 十一、结论

- 模块入口附录只回答“是否可见、是否可进、有哪些合法入口、默认进入模式是什么”。
- 模块正文、模块名单和实体正文继续转交 [附录-模块正文配置与模块名单规范](附录-模块正文配置与模块名单规范.md) 及其相邻专题。
- 入口切换顺序、恢复后流程与验证直达链路继续只留在业务链路正文。

## 十二、本文不负责的事项

- 不负责首页标题、首页预览展示包和公共导航字段。
- 不负责 `module.yaml`、`role_roster.yaml`、`enemy_roster.yaml`、`map_roster.yaml` 的正文契约。
- 不负责角色固定武器、技能实体、投射物、状态效果和关卡波次正文。
