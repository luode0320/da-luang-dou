# 存档与资产生命周期规范

## 目标

本附录用于把“每模块单一自动存档”“继续上次 / 开始新游戏”“长期资产保留”“模块内临时资产清理”和“写入时机/清理时机”写成长期架构规则，避免后续实现时出现以下偏差：

- 把模块内临时道具错误做成跨模块永久资产
- 失败时错误清空已解锁内容或永久道具
- `继续上次` 与 `开始新游戏` 的语义不一致
- 存档只保存战斗快照，却漏掉模块解锁和长期资产
- 配置变更后无法判断哪些字段影响存档兼容

## 一、核心结论

### 1. 单一自动存档

- 每个模块只保留一个自动存档状态。
- 不提供多个存档槽。
- 玩家默认只看到 `继续上次` 与 `开始新游戏` 两个入口。

### 2. 长期资产与临时资产必须分层

- 普通、紫色、临时金色道具属于模块内临时资产。
- 永久金色道具、模块解锁状态、已通关关卡、已解锁角色等属于长期资产。
- 模块内临时资产在模块结束后清理。
- 长期资产在失败或退出后仍然保留。

### 3. 失败保护不等于无限回档

- 失败不清除长期资产。
- 失败后允许返回当前模块的继续入口。
- 失败不应把当前模块外的长期进度回滚成更早状态。

## 二、生命周期对象

长期架构上，至少区分四类对象：

| 对象 | 生命周期 | 典型内容 | 模块结束后 | 失败后 |
| --- | --- | --- | --- | --- |
| 模块自动存档 | 模块级 | 当前节点、当前关卡、局内恢复信息 | 更新或重置 | 保留 |
| 长期进度 | 跨模块长期存在 | 模块解锁、关卡通关、角色解锁 | 保留 | 保留 |
| 长期资产 | 跨模块长期存在 | 永久金色道具、外部携带位 | 保留 | 保留 |
| 模块临时资产 | 仅当前模块有效 | 普通、紫色、临时金色道具、局内金币状态 | 清理 | 可保留到本模块继续入口，模块结束后清理 |

## 三、继续上次 / 开始新游戏 语义

### 继续上次

`继续上次` 的目标是恢复当前模块的唯一自动存档入口，而不是回到任意历史节点。

恢复内容至少包括：

- 当前模块 ID
- 当前可进入节点
- 当前模块内临时资产状态
- 当前局面恢复所需的最小运行信息
- 当前长期资产快照引用
- 具体“恢复后先进入哪个流程节点、随后如何继续推进”继续转交 [6-业务链路-模块闯关与进度推进](6-业务链路-模块闯关与进度推进.md)。

### 开始新游戏

`开始新游戏` 的目标是重置当前模块的模块级进度，而不是清空长期资产。

重置后应满足：

- 当前模块自动存档被重置到模块起点
- 当前模块内临时资产被清空
- 当前模块局内金币与局内抽取状态被清空
- 长期资产、永久金色道具、已解锁模块和已通关关卡仍保留
- 本附录只定义“重置什么 / 保留什么”；真正的入口切换和结算后跳转顺序继续转交业务链路正文。

## 四、建议的存档分层模型

以下是长期建议的逻辑分层模型，用于指导后续实现；不代表当前仓库已经存在对应代码文件。

### 1. 存档总索引

负责记录：

- 当前存档结构版本
- 当前模块列表
- 每个模块是否存在自动存档
- 当前长期资产版本

### 2. 模块自动存档

每个模块一份记录，负责保存：

- `module_id`
- 当前节点
- 当前关卡
- 当前难度
- 当前角色和外部道具搭配
- 当前模块内临时资产
- 最近一次写入时间

### 3. 长期进度记录

负责保存：

- 已解锁模块
- 已通关节点或关卡
- 模块完成状态
- 关键引导状态

### 4. 长期资产记录

负责保存：

- 永久金色道具列表
- 外部道具携带位状态
- 需要跨模块保留的角色或系统级资产

## 五、建议的最小存档字段

### 模块自动存档最小字段

| 字段 | 作用 |
| --- | --- |
| `save_schema_version` | 存档结构版本 |
| `module_id` | 当前模块稳定 ID |
| `current_node_id` | 当前节点 |
| `current_level_id` | 当前关卡 |
| `difficulty_id` | 当前难度 |
| `role_id` | 当前角色 |
| `equipped_external_item_ids` | 当前外部永久金色道具搭配 |
| `module_temp_item_ids` | 当前模块内临时道具 |
| `module_temp_currency` | 当前模块内金币或局内货币 |
| `last_release_version` | 对应配置发布版本 |
| `last_saved_at` | 最近保存时间 |

### 长期资产最小字段

| 字段 | 作用 |
| --- | --- |
| `owned_permanent_item_ids` | 已获得永久金色道具 |
| `unlocked_module_ids` | 已解锁模块 |
| `cleared_level_ids` | 已通关关卡 |
| `unlocked_role_ids` | 已解锁角色 |
| `profile_progress_version` | 长期进度结构版本 |

## 六、写入时机建议

至少建议在以下时机写入自动存档或长期进度：

1. 进入模块后完成模块恢复
2. 进入新节点或新关卡时
3. 通关当前关卡时
4. 解锁新节点或新模块时
5. 获得永久金色道具时
6. 主动退出模块返回首页时

不建议在以下时机做全量持久化：

- 每帧
- 每次攻击
- 每次受击
- 每次掉落判定瞬间

## 七、失败、退出与完成后的保留/清理规则

### 失败

- 保留长期资产
- 保留模块自动存档入口
- 不清除已解锁节点或模块
- 不删除永久金色道具
- 本附录不定义失败后页面如何跳转，只定义失败后哪些状态继续存在。

### 模块完成

- 更新长期进度
- 结算永久奖励
- 清理当前模块临时资产
- 把模块自动存档推进到新的合法继续入口或模块起点
- 模块完成后的“进入下一节点 / 回到模块入口 / 回到首页”顺序，继续转交业务链路正文。

### 模块中途退出

- 保留当前模块自动存档
- 保留当前模块临时资产，供同模块继续上次恢复
- 不将“中途退出”解释为“模块结束”

## 八、与配置发布的关系

- 存档永远引用稳定 ID，不引用临时显示名。
- 存档恢复时必须绑定对应的发布版本或兼容映射。
- 后台发布前必须检查实体 ID 删除、字段更名、模块节点变化是否破坏现有自动存档。
- `compatibility.yaml` 中的兼容规则必须覆盖存档读取所需的字段别名和 ID 重定向。

## 九、与资产分层的关系

### 长期资产

包括但不限于：

- 永久金色道具
- 模块解锁
- 角色解锁
- 已通关关卡记录

### 模块临时资产

包括但不限于：

- 普通道具
- 紫色道具
- 临时金色道具
- 当前模块局内金币
- 只在本模块成立的局内效果

## 十、测试与调试要求

阶段 7 至少应覆盖以下验证：

- 旧自动存档在兼容范围内可继续读取
- 失败后长期资产未被清空
- `开始新游戏` 只重置当前模块进度，不清空永久资产
- 模块结束后模块内临时资产被清理
- 调试入口支持指定模块、关卡、角色、外部道具和存档状态启动

## 十二、结论

后续实现里，最重要的不是“先把存档写出来”，而是先保持下面这条边界稳定：

- 模块自动存档只负责当前模块继续入口
- 长期进度和长期资产不因失败被清空
- 模块内临时资产只在模块生命周期内有效

只要这条边界先写硬，后续 `继续上次`、`开始新游戏`、通关结算、失败保护和存档兼容都会更容易统一；而默认 20 关、默认 60 秒、Boss 节点和节点关系校验继续只留给关卡节点专题。
